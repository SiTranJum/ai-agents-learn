## 第一部分：LLM 的知识局限

### 1.1 LLM 知道很多，但不知道三件事

```
┌─────────────────────────────────────────────────────┐
│              LLM 不知道的三件事                       │
├─────────────────┬───────────────┬───────────────────┤
│  时效性知识       │  私有知识       │  精确数据          │
├─────────────────┼───────────────┼───────────────────┤
│ "最新的营养学     │ "你公司的内部   │ "100g 鸡胸肉      │
│  研究说什么？"    │  健康食谱？"    │  到底多少卡？"     │
├─────────────────┼───────────────┼───────────────────┤
│ 训练数据有截止日期 │ 从未见过你的数据│ LLM 会估算，      │
│ 之后的事不知道    │ 当然不知道      │ 但不一定准确       │
└─────────────────┴───────────────┴───────────────────┘
```

### 1.2 幻觉问题

LLM 不知道答案时，不会说"我不知道"，而是**自信地编造一个**。

```python
# 问 LLM 一个具体的营养数据
user: "100g 紫薯的嘌呤含量是多少？"

# LLM 可能回答：
"100g 紫薯的嘌呤含量约为 15mg，属于低嘌呤食物。"
# ↑ 这个数字可能是编的！LLM 不会去查数据库

# 对于健康管家来说，这很危险
# 痛风患者如果根据错误的嘌呤数据选择食物，可能导致发作
```

**Java 类比**：就像一个 Service 层方法，没有查数据库就直接 return 了一个看起来合理的值。编译不报错，运行不报错，但数据是错的。

### 1.3 解决思路

```
方案 1：微调（Fine-tuning）
  把知识"塞进"模型 → 成本高、更新慢、不灵活
  类比：把数据 hardcode 在代码里

方案 2：RAG（检索增强生成）
  用的时候再去查 → 成本低、实时更新、灵活
  类比：代码里查数据库，用最新的数据

→ 绝大多数场景选 RAG
```

## 第二部分：RAG 的核心原理

### 2.1 一句话解释

**RAG = 开卷考试**

```
闭卷考试（普通 LLM）：
  老师提问 → 学生凭记忆回答 → 可能记错

开卷考试（RAG）：
  老师提问 → 学生先翻书找到相关内容 → 基于书本内容回答 → 更准确
```

### 2.2 RAG 的全称

**R**etrieval **A**ugmented **G**eneration（检索增强生成）

```
Retrieval（检索）：从知识库中找到相关的内容
Augmented（增强）：把检索到的内容塞进 Prompt
Generation（生成）：LLM 基于增强后的 Prompt 生成回答
```

### 2.3 RAG 的本质

RAG 本质上就是**在调用 LLM 之前，先查一遍资料，把资料塞进 Prompt**。

```python
# 没有 RAG
messages = [
    {"role": "user", "content": "100g 鸡胸肉的嘌呤含量是多少？"}
]
# LLM 凭记忆回答，可能编造数据

# 有 RAG
# 第 1 步：从知识库检索
docs = search_knowledge_base("鸡胸肉 嘌呤")
# → "鸡胸肉每100g嘌呤含量约为141mg，属于中等嘌呤食物。——《中国食物成分表》"

# 第 2 步：把检索结果塞进 Prompt
messages = [
    {"role": "system", "content": "根据以下参考资料回答用户问题。如果资料中没有相关信息，请明确说明。"},
    {"role": "user", "content": (
        "参考资料：\n"
        "鸡胸肉每100g嘌呤含量约为141mg，属于中等嘌呤食物。——《中国食物成分表》\n\n"
        "用户问题：100g 鸡胸肉的嘌呤含量是多少？"
    )}
]
# LLM 基于真实数据回答，准确！
```

**就这么简单。** RAG 不是什么神秘的技术，本质就是"先查再答"。

### 2.4 Java 类比

```java
// 没有 RAG
public String answer(String question) {
    return llm.generate(question);  // 凭记忆回答
}

// 有 RAG
public String answer(String question) {
    // 1. 检索相关文档（类似查数据库）
    List<Document> docs = knowledgeBase.search(question);
    
    // 2. 拼入 Prompt（类似拼 SQL 结果到模板）
    String context = docs.stream()
        .map(Document::getContent)
        .collect(Collectors.joining("\n"));
    
    String prompt = "参考资料：\n" + context + "\n\n问题：" + question;
    
    // 3. LLM 基于资料回答
    return llm.generate(prompt);
}
```

## 第三部分：关键问题 — 怎么检索？

### 3.1 传统检索 vs 语义检索

**传统检索（关键词匹配）**：
```
用户问："吃什么对关节好？"
关键词搜索："关节" → 
  ✅ "关节炎的饮食注意事项"
  ❌ "富含 omega-3 的食物有助于减轻炎症"  ← 没有"关节"这个词，搜不到！
```

**语义检索（Embedding）**：
```
用户问："吃什么对关节好？"
语义搜索：理解"关节好" ≈ "减轻炎症" ≈ "骨骼健康"
  ✅ "关节炎的饮食注意事项"
  ✅ "富含 omega-3 的食物有助于减轻炎症"  ← 语义相关，能搜到！
```

### 3.2 Embedding 是什么

Embedding = 把文本变成一串数字（向量），让计算机能理解语义。

```python
# 文本 → 向量
"鸡蛋" → [0.12, -0.34, 0.78, ..., 0.56]  # 1536 维向量
"鸡蛋的营养" → [0.15, -0.31, 0.75, ..., 0.58]  # 和"鸡蛋"很接近
"汽车" → [0.89, 0.23, -0.45, ..., -0.12]  # 和"鸡蛋"很远
```

**核心性质**：语义相近的文本，向量也相近。

```
"鸡蛋" 和 "鸡蛋的营养" → 向量距离近（相似度 0.95）
"鸡蛋" 和 "汽车"       → 向量距离远（相似度 0.12）
```

**Java 类比**：
- 传统检索 = `SELECT * FROM docs WHERE content LIKE '%关键词%'`
- 语义检索 = 把文本变成向量后，用向量距离排序（`ORDER BY vector_distance`）

### 3.3 向量数据库

普通数据库存文本，向量数据库存向量，并且**擅长快速找到最相似的向量**。

```
普通数据库（MySQL）：
  SELECT * FROM foods WHERE name = '鸡蛋'  ← 精确匹配

向量数据库（Milvus）：
  search(query_vector, top_k=5)  ← 找到最相似的 5 个向量
```

**常见向量数据库**：
| 数据库 | 特点 |
|--------|------|
| Milvus | 开源，性能好，我们项目用这个 |
| Pinecone | 云服务，简单易用 |
| Chroma | 轻量级，适合开发阶段 |
| pgvector | PostgreSQL 插件，不用额外部署 |

## 第四部分：RAG 完整流程

### 4.1 两个阶段

```
┌─────────────────────────────────────────────────────┐
│                    离线阶段（准备知识库）               │
│                                                     │
│  文档 → 分块(Chunking) → Embedding → 存入向量数据库  │
│                                                     │
│  "中国食物       "鸡胸肉每100g     [0.12, -0.34,    │
│   成分表.pdf"     含蛋白质31g,       0.78, ...]      │
│                   热量133卡..."                      │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│                    在线阶段（用户提问）                │
│                                                     │
│  用户提问 → Embedding → 向量检索 → 拼入Prompt → LLM  │
│                                                     │
│  "鸡胸肉    [0.15, -0.31,  找到最相似    "参考资料：  │
│   多少卡？"   0.75, ...]    的文档块      鸡胸肉..."  │
└─────────────────────────────────────────────────────┘
```

### 4.2 离线阶段详解

**第 1 步：文档分块（Chunking）**

大文档不能整个塞进向量数据库，需要切成小块。

```
原始文档（中国食物成分表）：
"鸡胸肉每100g含蛋白质31g，脂肪1.2g，热量133卡。
 鸡蛋每个约50g，含蛋白质6g，脂肪5g，热量70卡。
 牛肉每100g含蛋白质26g，脂肪10g，热量250卡。
 ..."

分块后：
Chunk 1: "鸡胸肉每100g含蛋白质31g，脂肪1.2g，热量133卡。"
Chunk 2: "鸡蛋每个约50g，含蛋白质6g，脂肪5g，热量70卡。"
Chunk 3: "牛肉每100g含蛋白质26g，脂肪10g，热量250卡。"
```

**第 2 步：Embedding**

每个块转成向量。

```python
# 调用 Embedding API（和调用 LLM 类似，只是换了个接口）
# OpenAI SDK 的 embeddings.create() 方法
response = client.embeddings.create(
    model="text-embedding-v3",  # Embedding 模型
    input="鸡胸肉每100g含蛋白质31g，脂肪1.2g，热量133卡"
)
vector = response.data[0].embedding  # → [0.12, -0.34, 0.78, ...]
```

**第 3 步：存入向量数据库**

```python
# 存入 Milvus（伪代码）
milvus.insert(
    id=1,
    vector=[0.12, -0.34, 0.78, ...],
    text="鸡胸肉每100g含蛋白质31g，脂肪1.2g，热量133卡"
)
```

### 4.3 在线阶段详解

**第 1 步：用户提问 → Embedding**

```python
query = "鸡胸肉多少卡？"
query_vector = embed(query)  # → [0.15, -0.31, 0.75, ...]
```

**第 2 步：向量检索**

```python
# 在向量数据库中找最相似的文档
results = milvus.search(query_vector, top_k=3)
# → [
#     "鸡胸肉每100g含蛋白质31g，脂肪1.2g，热量133卡",  ← 最相似
#     "鸡蛋每个约50g，含蛋白质6g，脂肪5g，热量70卡",
#     "牛肉每100g含蛋白质26g，脂肪10g，热量250卡"
#   ]
```

**第 3 步：拼入 Prompt**

```python
context = "\n".join(results)
messages = [
    {"role": "system", "content": (
        "你是健康管家。根据以下参考资料回答用户问题。"
        "如果资料中没有相关信息，请明确说明，不要编造。"
    )},
    {"role": "user", "content": f"参考资料：\n{context}\n\n问题：{query}"}
]
```

**第 4 步：LLM 生成回答**

```python
response = client.chat.completions.create(
    model="deepseek-chat",
    messages=messages
)
# → "鸡胸肉每 100g 热量约 133 卡，蛋白质含量高达 31g，
#     脂肪仅 1.2g，是减脂期非常理想的蛋白质来源。"
```

### 4.4 Java 类比完整版

```java
// RAG 就像这样一个 Service
@Service
public class RAGService {
    @Autowired VectorDB vectorDB;   // 类似 Repository
    @Autowired LLMClient llm;       // 类似 RestTemplate

    public String answer(String question) {
        // 1. Embedding（把问题变成向量）
        float[] queryVector = llm.embed(question);
        
        // 2. 检索（在向量数据库中搜索，类似 findByXxx）
        List<Document> docs = vectorDB.search(queryVector, 3);
        
        // 3. 拼装 Prompt（类似拼 DTO）
        String context = docs.stream()
            .map(Document::getText)
            .collect(joining("\n"));
        
        // 4. LLM 生成（类似调外部接口）
        return llm.chat("参考资料：" + context + "\n问题：" + question);
    }
}
```

## 第五部分：健康管家的 RAG 场景

### 5.1 三个知识库

```
┌──────────────────────────────────────────────────┐
│             健康管家的三个知识库                     │
├──────────────────┬───────────────┬───────────────┤
│  食物营养知识库    │  健康建议知识库 │  用户记忆库    │
├──────────────────┼───────────────┼───────────────┤
│ 中国食物成分表    │ 营养学指南     │ "用户对海鲜过敏"│
│ 常见菜品热量      │ 减肥/增肌建议  │ "用户在减脂期"  │
│ 食物嘌呤含量      │ 各类疾病饮食   │ "用户不吃辣"   │
│ 食物 GI 值       │ 运动营养指导   │ 用户的历史饮食  │
├──────────────────┼───────────────┼───────────────┤
│ 数据来源：权威文献 │ 数据来源：专业  │ 数据来源：对话  │
│ 更新频率：低      │ 指南和教材     │ 中提取         │
│ 精确度要求：高    │ 更新频率：中   │ 更新频率：高    │
└──────────────────┴───────────────┴───────────────┘
```

### 5.2 实际使用场景

**场景 1：食物营养查询**
```
用户："紫薯的嘌呤高吗？痛风能吃吗？"

RAG 流程：
1. 检索 → 找到"紫薯每100g嘌呤含量约12mg，属于低嘌呤食物"
2. 拼入 Prompt
3. LLM 回答："紫薯嘌呤含量很低（12mg/100g），痛风患者可以放心食用。"

没有 RAG：
LLM 回答："紫薯嘌呤含量约15mg..."  ← 数字可能是编的
```

**场景 2：个性化建议（知识库 + 记忆库联合检索）**
```
用户："中午吃什么好？"

RAG 流程：
1. 检索记忆库 → "用户对海鲜过敏""用户在减脂期""最近常吃鸡胸肉"
2. 检索知识库 → "减脂期午餐建议 500-600 卡"
3. 拼入 Prompt
4. LLM 综合两个来源给出个性化建议
```

### 5.3 RAG vs Tool Use

```
Tool Use（课程 3）：
  "查询鸡蛋的热量" → 调用 food_api.query("鸡蛋") → 返回精确数据
  适合：有明确 API 可调用的场景

RAG：
  "吃什么对关节好？" → 检索营养学文档 → 找到相关段落 → LLM 总结
  适合：需要从大量非结构化文档中找答案的场景
```

两者不冲突，健康管家会同时用到：
- 精确营养数据 → Tool Use（查 API）
- 健康知识问答 → RAG（查知识库）

## 小结

### 核心概念

1. **RAG = 开卷考试**：先检索相关知识，再让 LLM 基于知识回答
2. **解决三个问题**：知识时效性、私有数据、减少幻觉
3. **Embedding**：把文本变成向量，语义相近的文本向量距离近
4. **两个阶段**：离线（文档→分块→向量→存储）、在线（提问→检索→拼Prompt→回答）
5. **本质很简单**：就是在调用 LLM 之前，先查一遍资料塞进 Prompt

### 与已学知识的关系

```
RAG 在 Agent 技术体系中的位置：

Planning ──── 能力层
ReAct / Multi-Step ──── 模式层
CoT ──── 技巧层
Tool Use ──── 基础设施（精确数据查询）
Memory ──── 基础设施（用户记忆）
RAG ──── 基础设施（外部知识检索）← 新增！
```
