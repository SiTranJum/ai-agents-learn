# 课程 4：什么是 AI Agent？

## 第一部分：从对比开始

### 普通 LLM 调用

你前三节课写的代码，大部分是这种模式：

```
用户提问 → LLM 回答 → 结束
```

特点：
- **你控制一切**：你决定什么时候调用 LLM、传什么参数、怎么处理结果
- **单轮决策**：LLM 回答完就结束，不会自己决定"我还需要做什么"
- **无记忆**：每次调用都是独立的，不知道之前发生了什么
- **你是大脑**：你（程序员）写了所有逻辑，LLM 只是一个被调用的工具

### AI Agent

```
用户提问 → Agent 分析意图 → Agent 决定行动 → 执行 → 检查结果 → 决定下一步 → ... → 最终回答
```

特点：
- **Agent 自己控制流程**：它决定调用什么工具、调用几次、什么时候结束
- **多步决策**：一个任务可能需要多步，Agent 自己规划和执行
- **有记忆**：知道之前发生了什么，能基于历史做决策
- **LLM 是大脑**：LLM 负责思考和决策，你只提供工具和环境

---

## 第二部分：Agent 的定义

### 一句话定义

> **AI Agent = 一个能自主感知环境、做出决策、采取行动来完成目标的 AI 系统**

### 类比理解

| 角色 | 普通 LLM 调用 | AI Agent |
|------|-------------|----------|
| **Java 类比** | 一个 Service 方法 | 一个有状态的微服务，能自主处理请求 |
| **现实类比** | 翻译器：你给文本，它翻译 | 私人助理：你给目标，它自己规划和执行 |
| **游戏类比** | NPC 的一句固定台词 | 有 AI 的 NPC，能自主行动 |

### Agent 的核心公式

```
Agent = LLM（大脑）
      + 感知（Perception）—— 理解输入
      + 记忆（Memory）—— 记住历史
      + 工具（Tool Use）—— 与外部交互
      + 规划（Planning）—— 拆解任务
```

用 Java 架构类比：

```
Agent ≈ 一个 Spring Boot 微服务

LLM          = Controller（接收请求、做决策）
感知          = RequestParser（解析请求，理解意图）
记忆          = Redis/数据库（存储状态和历史）
工具          = Service 层（执行具体业务逻辑）
规划          = 工作流引擎（编排多步任务）
```

## 第三部分：四大核心能力

### 1. 感知（Perception）

Agent 需要理解用户到底想做什么。

```
用户："我中午吃了一碗红烧肉和米饭"

感知需要理解：
- 意图：记录饮食
- 时间：中午（lunch）
- 食物：红烧肉、米饭
- 数量：一碗（需要估算克数）
```

这主要靠 **Prompt Engineering**（课程 02）来实现。

### 2. 记忆（Memory）

Agent 需要记住历史信息，做出更好的决策。

```
用户上周说："我在减肥，目标体重 65kg"
用户今天说："我中午吃了炸鸡"

有记忆的 Agent：
"炸鸡热量较高（约 500 千卡），考虑到你的减肥目标，建议晚餐吃清淡些。"

没记忆的 Agent：
"已记录：炸鸡。"
```

这是课程 06 会深入讲的内容。

### 3. 工具（Tool Use）

Agent 需要与外部世界交互，获取信息、执行操作。

```
用户："帮我查一下红烧肉的热量"

Agent 的工具调用：
1. 调用 search_food("红烧肉") → 找到食物 ID
2. 调用 get_nutrition(food_id) → 获取营养数据
3. 基于结果回答用户
```

这是课程 03 已经学过的内容。

### 4. 规划（Planning）

Agent 需要把复杂任务拆解成多个步骤。

```
用户："帮我制定一个减肥计划"

Agent 的规划：
1. 查看用户健康档案（身高、体重、目标）
2. 计算每日所需热量（TDEE）
3. 制定热量摄入目标
4. 根据用户饮食偏好生成食谱建议
5. 制定运动计划
6. 设置提醒
```

这是课程 07 会深入讲的内容。

## 第四部分：代码对比

### 场景：用户问"我今天吃了什么，还能吃多少"

这个场景需要：
1. 查询今天的饮食记录
2. 计算已摄入热量
3. 查询用户的每日热量目标
4. 计算剩余可摄入热量
5. 给出建议

### 普通 LLM 调用的实现

程序员需要手动编排所有步骤：

```python
# 你（程序员）控制每一步
records = query_meals(date="today")        # 第 1 步：你决定查询
calories = calculate_calories(records)      # 第 2 步：你决定计算
target = get_user_target(user_id)          # 第 3 步：你决定查目标
remaining = target - calories              # 第 4 步：你决定计算差值

# 最后才调用 LLM 生成回复
response = llm.chat(f"用户今天吃了 {records}，已摄入 {calories} 千卡，目标 {target}，剩余 {remaining}，请给出建议")
```

问题：
- 流程是你写死的，LLM 只是最后的"文案生成器"
- 如果需求变了（比如用户问的是"这周吃了什么"），你需要改代码
- LLM 没有参与决策

### Agent 的实现

Agent 自主决策每一步：

```python
# 你只定义工具和目标，Agent 自己决定怎么做
agent.tools = [query_meals, calculate_calories, get_user_target, ...]
agent.run("我今天吃了什么，还能吃多少")

# Agent 内部的思考过程：
# 思考：用户想知道今天的饮食和剩余额度
# 行动：调用 query_meals(date="today")
# 观察：返回了 [红烧肉, 米饭, 苹果]
# 思考：需要计算总热量
# 行动：调用 calculate_calories([红烧肉, 米饭, 苹果])
# 观察：总计 750 千卡
# 思考：需要知道用户的目标
# 行动：调用 get_user_target(user_id)
# 观察：目标 1800 千卡
# 思考：剩余 1050 千卡，可以给建议了
# 回答：你今天吃了...还可以摄入 1050 千卡...建议...
```

优势：
- Agent 自己决定调用什么、调用几次
- 换个问题（"这周吃了什么"），不用改代码，Agent 自己调整
- LLM 参与了整个决策过程

## 第五部分：Agent 的本质区别

| 维度 | 普通 LLM 调用 | AI Agent |
|------|-------------|----------|
| **控制权** | 程序员控制流程 | Agent 自主控制 |
| **决策能力** | 无，按代码逻辑执行 | 有，LLM 做决策 |
| **灵活性** | 低，逻辑写死 | 高，能应对新场景 |
| **工具使用** | 程序员决定调用什么 | Agent 决定调用什么 |
| **记忆** | 无（或程序员手动管理） | 有，Agent 自己管理 |
| **错误处理** | 程序员写 try-catch | Agent 自己判断并重试 |
| **适用场景** | 简单、固定的任务 | 复杂、灵活的任务 |

### 一句话总结

> **普通 LLM 调用：你是大脑，LLM 是工具**
> **AI Agent：LLM 是大脑，你提供工具和环境**
