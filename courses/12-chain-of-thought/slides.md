# 课程 12：Chain of Thought（CoT）

---

## 一、什么是 Chain of Thought？

### 1.1 一句话解释

**CoT = 让 LLM 展示推理过程，而不是直接给答案。**

**Java 类比**：
- 无 CoT = `return result;`（直接返回结果，不知道怎么算的）
- 有 CoT = 方法里写了详细的日志 log，每一步都能追踪

### 1.2 为什么有效？

LLM 本质是"下一个 token 预测"。当它直接回答时，容易跳过中间步骤导致错误。

**强制它一步步写出来 → 每一步的输出成为下一步的输入 → 推理链更完整**

```
无 CoT：问题 → 答案（中间推理在"黑盒"中，容易出错）

有 CoT：问题 → 步骤1 → 步骤2 → 步骤3 → 答案（每步都显式推理）
```

### 1.3 适用场景

| 场景 | 是否需要 CoT |
|------|-------------|
| 简单问候（"你好"） | ❌ 不需要 |
| 事实查询（"鸡蛋多少卡"） | ❌ 不需要 |
| 数学计算（"今天摄入总热量"） | ✅ 需要 |
| 多条件判断（"饮食是否合理"） | ✅ 需要 |
| 方案制定（"减肥计划"） | ✅ 需要 |

**规律**：任务越复杂、需要的推理步骤越多，CoT 的效果越好。

## 二、CoT 的三种用法

### 2.1 Zero-shot CoT

**最简单的用法**：在 Prompt 末尾加一句"让我们一步步思考"。

```python
# 无 CoT
prompt = "我今天吃了 2 个鸡蛋和一碗米饭，目标 1500 卡，还能吃多少？"

# 有 Zero-shot CoT
prompt = "我今天吃了 2 个鸡蛋和一碗米饭，目标 1500 卡，还能吃多少？请一步步推理。"
```

**Java 类比**：就像在方法注释里写 `// 请逐步计算`，提醒自己不要跳步。

**优点**：极其简单，一句话就能用  
**缺点**：推理格式不可控，质量取决于 LLM 自身能力

### 2.2 Few-shot CoT

**给出推理示例**，让 LLM 模仿你的推理方式。

```python
system_prompt = """你是健康管家 AI。回答问题时请按以下格式逐步推理：

示例：
问题：我吃了 3 个苹果，每个 52 卡，目标 2000 卡，还能吃多少？
推理：
1. 苹果热量：3 × 52 = 156 卡
2. 已摄入：156 卡
3. 剩余额度：2000 - 156 = 1844 卡
答案：您还可以摄入约 1844 卡路里。
"""
```

**Java 类比**：类似写单元测试 — 给出输入和期望输出，让 LLM 按照模式执行。

**优点**：推理格式可控，质量稳定  
**缺点**：需要手动编写示例，占用上下文窗口

### 2.3 Self-Consistency CoT

**多次推理，取一致性最高的结果。**

```
推理 1（temperature=0.7）：总热量 340 卡，剩余 1160 卡
推理 2（temperature=0.7）：总热量 340 卡，剩余 1160 卡
推理 3（temperature=0.7）：总热量 350 卡，剩余 1150 卡

投票结果：1160 卡（2/3 一致）
```

**Java 类比**：类似分布式系统的"多数投票"（Quorum），多个节点计算同一个结果，取多数一致的。

**优点**：结果更可靠  
**缺点**：需要多次 API 调用，成本高

## 三、CoT vs ReAct

| 维度 | CoT | ReAct |
|------|-----|-------|
| **核心目标** | 提升推理质量 | 结合推理与工具调用 |
| **输出内容** | 思考步骤 + 答案 | Thought + Action + Observation |
| **是否调用工具** | 不一定 | 一定会 |
| **适用场景** | 纯推理任务 | 需要外部信息的任务 |
| **实现复杂度** | 低（改 Prompt 即可） | 中（需要解析和循环） |

**关系**：ReAct 的 Thought 部分就是 CoT。

```
ReAct = CoT + Tool Use

Thought: [这里就是 CoT]      ← 推理过程
Action: query_nutrition(...)  ← 工具调用
Observation: {...}            ← 工具结果
```

**怎么选？**
- 纯推理（不需要查数据）→ 用 CoT
- 需要查数据 + 推理 → 用 ReAct
- 两者可以结合使用

## 四、在健康管家中应用 CoT

### 4.1 场景：饮食分析

**无 CoT**：
```
用户：我今天早餐吃了 2 个鸡蛋，午餐吃了一碗米饭和鸡胸肉 100g，
     晚餐吃了一个苹果。我的目标是 1500 卡，今天达标了吗？

AI：您今天摄入约 500 卡，远低于目标 1500 卡，建议多吃一些。
```

回答不够精确，也没有推理过程。

**有 CoT**：
```
用户：（同上）

AI：让我逐步计算：

1. 早餐：2 个鸡蛋 = 2 × 70 = 140 卡
2. 午餐：
   - 一碗米饭 = 200 卡
   - 鸡胸肉 100g = 165 卡
   - 午餐小计 = 365 卡
3. 晚餐：1 个苹果 = 52 卡
4. 今日总摄入 = 140 + 365 + 52 = 557 卡
5. 目标 1500 卡，差距 = 1500 - 557 = 943 卡

结论：您今天摄入 557 卡，离目标还差 943 卡，摄入明显不足。
建议晚上再补充一顿，比如一碗米饭 + 蔬菜 + 蛋白质。
```

### 4.2 场景：健康评估

```
用户：我身高 175cm，体重 80kg，每天吃 2000 卡，想减到 70kg，
     大概需要多久？

AI：让我逐步分析：

1. 当前 BMI = 80 / (1.75²) = 26.1（偏胖）
2. 目标体重 70kg，需要减 10kg
3. 减 1kg 脂肪约需消耗 7700 卡
4. 总共需消耗 10 × 7700 = 77000 卡
5. 基础代谢率（男性）≈ 1750 卡/天
6. 每日摄入 2000 卡，与基础代谢差不多
7. 如果加上日常活动消耗约 300 卡/天，每日热量缺口约 50 卡
8. 按此速度：77000 / 50 = 1540 天（约 4 年）

结论：按当前饮食，减肥速度非常慢。
建议：将每日摄入降到 1500 卡，热量缺口增加到 550 卡/天，
约 140 天（约 5 个月）可以达到目标。
```

## 五、小结

### 5.1 关键概念

1. **CoT = 展示推理过程**：让 LLM 一步步推理，而不是直接给答案
2. **三种用法**：
   - Zero-shot：加一句"请一步步思考"
   - Few-shot：给出推理示例
   - Self-Consistency：多次推理取一致结果
3. **CoT vs ReAct**：CoT 侧重推理，ReAct = CoT + 工具调用

### 5.2 实践要点

1. **简单任务不需要 CoT**：事实查询、简单问候不需要
2. **复杂推理必须用 CoT**：数学计算、多条件判断、方案制定
3. **Few-shot 效果最稳定**：给出示例，LLM 更容易遵循格式
4. **CoT 和 ReAct 可以结合**：在 ReAct 的 Thought 中使用 CoT
