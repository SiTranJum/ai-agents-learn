### 层次总结

```
Planning ──── 能力层（Agent 怎么拆解任务）
   │
   ├── ReAct ─────── 模式层（LLM 自己决定每一步）
   ├── Multi-Step ── 模式层（开发者预先定义流程）
   │
   ├── CoT ───────── 技巧层（不是模式！是 Prompt 技巧，可嵌入任何模式）
   │
   ├── Tool Use ──── 基础设施（被所有模式共用的"手和脚"）
   └── Memory ────── 基础设施（被所有模式共用的"记忆"）
```

### Java 类比

| AI 概念 | Java 类比 |
|---------|----------|
| Planning | 架构设计（决定系统怎么拆分） |
| ReAct | 事件驱动架构（运行时动态决定流程） |
| Multi-Step | Spring Batch（预定义 Job 流程） |
| CoT | 日志/Debug 模式（让过程可见，帮助推理） |
| Tool Use | Spring Cloud Feign（调用外部服务的能力） |
| Memory | Redis + MySQL（数据存储层） |

### 关键区分

- **CoT 不是 Agent 模式**，是 Prompt 技巧，可以"撒"在任何地方
- **ReAct vs Multi-Step** 的区别：谁是指挥官。ReAct 是 LLM 指挥，Multi-Step 是开发者指挥
- **生产环境常用混合模式**：主流程用 Multi-Step（可控），子步骤用 ReAct（灵活）

## Multi-Step Agent 核心要点

- `Step` 类：定义步骤（名称、函数、依赖、错误策略）
- `MultiStepPlanner`：编排引擎，拓扑排序 → 按层执行
- `context`：步骤间的数据总线（不是记忆，流程结束就没了）
- 拓扑排序：BFS 算法，和 Maven/Spring 解析依赖顺序是同一个算法
- 错误处理四种策略：raise / skip / retry（指数退避）/ fallback

## 框架生态认知

- **LangGraph**：用图编排 Agent，支持循环，能统一表达 ReAct 和 Multi-Step
- **AutoGen**：微软的多 Agent 对话框架，Agent 之间自由讨论
- **CrewAI**：给 Agent 分角色和任务，简单易用
- **Spring AI**：Java 的 AI 集成层，能做基础 AI 功能，但 Agent 编排能力弱于 Python 生态
- 健康管家项目用 LangGraph 就够了（Supervisor 模式）

## Python 语法收获

- `self.xxx` = 实例属性（类比 Java 的 `this.xxx`），其他方法能访问
- 不带 `self` 的变量 = 局部变量，方法结束就没了
- 类体下直接写的变量 = 类变量（类比 Java 的 `static` 字段）
- `@staticmethod` = Java 的 `static` 方法
- `@classmethod` = Python 特有，第一个参数是类本身（`cls`）
