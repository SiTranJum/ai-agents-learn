## 第一部分：课程 16 的局限

### 1.1 回顾课程 16

课程 16 我们实现了基础 RAG：

```python
KNOWLEDGE_BASE = [
    "鸡胸肉每100g含蛋白质31g...",  # 手写的知识
    "鸡蛋一个约50g...",
]

# 直接向量化，存入 Chroma
```

**问题**：
1. 知识是手写的，实际项目中知识来自文档（PDF、Word、网页）
2. 每条知识长度不一，有的太长（超过 LLM 上下文），有的太短（信息不完整）
3. 没有元数据（不知道这条知识来自哪个文档、哪个章节）
4. 检索质量一般（只靠向量相似度，没有其他优化）

### 1.2 真实场景的需求

```
健康管家的知识来源：
  - 《中国居民膳食指南》PDF（200 页）
  - 《食物成分表》Excel（5000 行）
  - 营养学论文 10 篇
  - 公司内部健康食谱 Word 文档

需要：
  1. 自动加载这些文档
  2. 智能分块（不能太长也不能太短）
  3. 保留元数据（来源、章节、日期）
  4. 优化检索质量
```

## 第二部分：文档分块（Chunking）

### 2.1 为什么要分块

```
问题 1：文档太长，无法整个向量化
  《膳食指南》200 页 → 10 万字 → 无法一次性 Embedding

问题 2：检索粒度太粗
  用户问："鸡蛋的热量"
  如果整本书是一个 chunk，检索到的是整本书 → LLM 看不完

问题 3：LLM 上下文有限
  即使检索到了，200 页塞不进 Prompt（上下文窗口有限）
```

**解决方案**：把文档切成小块（chunk），每块独立向量化。

```
《膳食指南》200 页
  ↓ 分块
Chunk 1: "第一章 营养素基础知识。蛋白质是..."（500 字）
Chunk 2: "碳水化合物分为单糖、双糖..."（500 字）
Chunk 3: "脂肪分为饱和脂肪和不饱和脂肪..."（500 字）
...
Chunk 400: "附录：常见食物营养成分表..."（500 字）
```

### 2.2 分块策略

**策略 1：固定长度分块**

```python
def chunk_by_length(text, chunk_size=500, overlap=50):
    """
    按固定字符数分块
    
    chunk_size: 每块多少字
    overlap: 块之间重叠多少字（避免语义被截断）
    """
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start += chunk_size - overlap  # 重叠部分
    return chunks

text = "鸡蛋是优质蛋白来源。一个鸡蛋约50g，含蛋白质6g。蛋黄含胆固醇..."
chunks = chunk_by_length(text, chunk_size=20, overlap=5)
# → ["鸡蛋是优质蛋白来源。一个鸡蛋约", "鸡蛋约50g，含蛋白质6g。蛋", "质6g。蛋黄含胆固醇..."]
```

**优点**：简单
**缺点**：可能把一句话切断，语义不完整

**策略 2：按句子分块**

```python
def chunk_by_sentence(text, max_sentences=3):
    """
    按句子分块，每块包含 N 个句子
    """
    sentences = text.split('。')  # 简单按句号切分
    chunks = []
    for i in range(0, len(sentences), max_sentences):
        chunk = '。'.join(sentences[i:i+max_sentences])
        chunks.append(chunk)
    return chunks
```

**优点**：不会切断句子
**缺点**：句子长度不一，chunk 大小不均匀

**策略 3：递归分块（推荐）**

```python
# LangChain 的 RecursiveCharacterTextSplitter
# 按优先级尝试分隔符：\n\n → \n → 。 → ， → 空格 → 字符
# 尽量在段落、句子边界切分，实在不行才切字符
```

### 2.3 chunk_size 和 overlap 怎么选

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| chunk_size | 500-1000 字 | 太小信息不完整，太大检索不精确 |
| chunk_overlap | 50-200 字 | 避免关键信息被切断 |

**经验法则**：
- 技术文档、论文：chunk_size 较大（800-1000），保持逻辑完整
- 对话、问答：chunk_size 较小（300-500），快速定位
- 健康知识：500-800 字，平衡信息完整性和检索精度

### 2.4 overlap 的作用

```
没有 overlap：
  Chunk 1: "鸡蛋是优质蛋白来源"
  Chunk 2: "一个鸡蛋约50g"
  → 如果用户问"鸡蛋的蛋白质"，可能两个 chunk 都检索不到完整信息

有 overlap：
  Chunk 1: "鸡蛋是优质蛋白来源。一个鸡蛋约"
  Chunk 2: "一个鸡蛋约50g，含蛋白质6g"
  → 两个 chunk 都包含"鸡蛋"和"蛋白质"，检索更准确
```

## 第三部分：元数据管理

### 3.1 什么是元数据

元数据 = 关于数据的数据，描述这个 chunk 的来源、属性。

```python
chunk = {
    "text": "鸡蛋一个约50g，含蛋白质6g...",  # 内容
    "metadata": {                              # 元数据
        "source": "中国食物成分表.pdf",
        "page": 15,
        "category": "蛋类",
        "date": "2023-01-01"
    }
}
```

### 3.2 元数据的作用

**作用 1：溯源**

```
用户："鸡蛋的热量是多少？"
AI："根据《中国食物成分表》第 15 页，鸡蛋每个约 70 卡。"
     ↑ 来自元数据
```

**作用 2：过滤检索**

```python
# 只检索"蛋类"相关的知识
results = collection.query(
    query_embeddings=[query_vector],
    where={"category": "蛋类"},  # 元数据过滤
    n_results=3
)
```

**作用 3：时效性管理**

```python
# 只检索 2023 年后的知识（旧知识可能过时）
where={"date": {"$gte": "2023-01-01"}}
```

### 3.3 Chroma 中使用元数据

```python
collection.add(
    ids=["doc_0", "doc_1"],
    embeddings=[vector1, vector2],
    documents=["鸡蛋...", "鸡胸肉..."],
    metadatas=[                          # ← 元数据
        {"source": "成分表.pdf", "category": "蛋类"},
        {"source": "成分表.pdf", "category": "肉类"}
    ]
)

# 检索时过滤
results = collection.query(
    query_embeddings=[query_vector],
    where={"category": "蛋类"},          # ← 只检索蛋类
    n_results=3
)
```

## 第四部分：完整 RAG Pipeline

### 4.1 架构图

```
┌─────────────────────────────────────────────────────────┐
│                    离线阶段（构建知识库）                  │
│                                                         │
│  原始文档（PDF/Word/Markdown）                           │
│     ↓                                                   │
│  文档加载器（提取文本 + 元数据）                          │
│     ↓                                                   │
│  文本分块（RecursiveCharacterTextSplitter）              │
│     ↓                                                   │
│  批量 Embedding                                         │
│     ↓                                                   │
│  存入 Chroma（向量 + 原文 + 元数据）                      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    在线阶段（用户提问）                    │
│                                                         │
│  用户问题                                                │
│     ↓                                                   │
│  问题 Embedding                                         │
│     ↓                                                   │
│  Chroma 检索（向量相似度 + 元数据过滤）                    │
│     ↓                                                   │
│  （可选）重排序（Reranking）                              │
│     ↓                                                   │
│  拼入 Prompt（检索结果 + 元数据）                         │
│     ↓                                                   │
│  LLM 生成回答                                            │
└─────────────────────────────────────────────────────────┘
```

### 4.2 与课程 16 的对比

| 环节 | 课程 16 | 课程 17 |
|------|---------|---------|
| 知识来源 | 手写字符串 | 真实文档（PDF/Markdown） |
| 分块 | 无（每条知识就是一个 chunk） | 智能分块（RecursiveCharacterTextSplitter） |
| 元数据 | 无 | 有（来源、分类、日期） |
| 检索 | 纯向量相似度 | 向量 + 元数据过滤 |
| 溯源 | 无法溯源 | 可以告诉用户知识来自哪里 |

## 第五部分：检索优化

### 5.1 重排序（Reranking）

**问题**：向量检索可能不够精确。

```
用户问："痛风患者能吃豆腐吗？"

向量检索 top 3：
  1. "豆腐每100g热量约76卡..."（相似度 0.85）
  2. "痛风患者应限制高嘌呤食物..."（相似度 0.82）
  3. "豆制品是优质蛋白来源..."（相似度 0.80）

问题：第 1 条虽然相似度最高，但没有回答"痛风能不能吃"
```

**解决方案**：重排序

```
向量检索 top 10 → 用更精确的模型重新打分 → 取 top 3

重排序后：
  1. "痛风患者应限制高嘌呤食物..."（重排分 0.95）
  2. "豆腐嘌呤含量中等，痛风急性期避免..."（重排分 0.90）
  3. "豆腐每100g热量约76卡..."（重排分 0.75）
```

### 5.2 混合检索

**纯向量检索的局限**：

```
用户问："2023年的膳食指南"
向量检索：可能检索到 2020 年的指南（语义相似）
```

**混合检索 = 向量检索 + 关键词检索 + 元数据过滤**

```python
results = collection.query(
    query_embeddings=[query_vector],
    where={
        "$and": [
            {"date": {"$gte": "2023-01-01"}},  # 元数据过滤
            {"source": {"$contains": "膳食指南"}}  # 关键词过滤
        ]
    },
    n_results=3
)
```

### 5.3 查询改写

**问题**：用户的问题可能表达不清。

```
用户问："那个红色的水果热量多少"
→ 向量检索效果差（"红色的水果"太模糊）
```

**解决方案**：让 LLM 先改写问题

```python
# 第 1 步：让 LLM 改写问题
rewritten = llm.generate(
    "用户问：那个红色的水果热量多少。"
    "请改写成更明确的问题，便于检索。"
)
# → "苹果的热量是多少？"

# 第 2 步：用改写后的问题检索
results = search(rewritten)
```

## 小结

### 核心概念

1. **文档分块**：把长文档切成小块，每块独立向量化
2. **chunk_size**：500-1000 字，平衡信息完整性和检索精度
3. **chunk_overlap**：50-200 字，避免关键信息被切断
4. **元数据**：记录来源、分类、日期，用于溯源和过滤
5. **检索优化**：重排序、混合检索、查询改写

### 完整 Pipeline

```
文档 → 加载 → 分块 → Embedding → 存入 Chroma（带元数据）
用户提问 → Embedding → 检索（向量+元数据） → 拼 Prompt → LLM
```
