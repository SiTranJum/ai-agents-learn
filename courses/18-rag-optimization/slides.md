## 第一部分：分块策略对比

### 1.1 实验设计

同一份文档，用三种分块策略，对比检索效果。

```
策略 A：固定长度（200 字，无 overlap）
策略 B：固定长度（200 字，overlap 50）
策略 C：递归分块（按段落 → 句子边界切分）

同一个问题："痛风能吃豆腐吗？"
→ 看哪种策略检索到的结果最相关
```

### 1.2 关键结论（先说结论，代码里验证）

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 固定长度 | 简单、块大小均匀 | 可能切断句子 | 格式统一的文档 |
| 固定长度 + overlap | 减少信息丢失 | 数据膨胀、检索重复 | 对精度要求高 |
| 递归分块 | 保持语义完整 | 块大小不均匀 | 大多数场景（推荐） |

## 第二部分：Reranker 重排序

### 2.1 思路

我们不用专门的 Reranker 模型（需要额外部署），而是**用 LLM 做重排序**。

```
第 1 步：向量检索 top 5（粗筛）
第 2 步：把 5 条结果 + 用户问题发给 DeepSeek
第 3 步：让 DeepSeek 按相关性打分（0-10 分）
第 4 步：按分数重新排序，取 top 3
```

### 2.2 优缺点

```
优点：
  - 不需要额外模型，DeepSeek 就能做
  - 效果好（LLM 理解能力强）

缺点：
  - 多一次 API 调用，增加延迟和成本
  - 只适合 top_k 较小的场景（不能让 LLM 排 100 条）
```

### 2.3 什么时候用

```
知识库 < 100 条：不需要，向量检索够用
知识库 100-10000 条：可以考虑，提升明显
知识库 > 10000 条：建议用专用 Reranker 模型（如 bge-reranker）
```

## 第三部分：混合检索

### 3.1 BM25 关键词检索

BM25 是经典的关键词检索算法，Elasticsearch 默认就用它。

```
核心思想：
  一个词在文档中出现次数越多 → 越相关
  一个词在所有文档中都出现 → 不重要（类似"的""是"）

公式（简化版）：
  score = TF × IDF
  TF = 词在这篇文档中出现的频率
  IDF = log(总文档数 / 包含这个词的文档数)

例子：
  用户搜"痛风 豆腐"
  文档 A："豆腐嘌呤中等，痛风急性期避免" → TF(痛风)=1, TF(豆腐)=1 → 高分
  文档 B："豆腐每100g热量76卡"           → TF(痛风)=0, TF(豆腐)=1 → 低分
```

### 3.2 RRF 合并算法

```
RRF（Reciprocal Rank Fusion）：

score(doc) = Σ 1/(k + rank_i)

k = 60（常数，平滑排名差异）
rank_i = 文档在第 i 路检索中的排名

例子：
  文档 A：向量检索排第 1，关键词检索排第 3
  score = 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323

  文档 B：向量检索排第 5，关键词检索排第 1
  score = 1/(60+5) + 1/(60+1) = 0.0154 + 0.0164 = 0.0318

  文档 A 分数更高（两路都靠前）
```

## 第四部分：查询改写

### 4.1 为什么要改写

```
用户的问题经常不够明确：

"那个红色的水果" → "苹果"
"吃什么对关节好" → "哪些食物有助于缓解关节炎症"
"上次说的那个" → 需要结合对话历史改写
```

### 4.2 实现方式

```python
# 让 LLM 改写问题
rewrite_prompt = """
用户问了一个问题，请改写成更明确、更适合检索的形式。
只输出改写后的问题，不要解释。

原始问题：{query}
改写后：
"""
```

### 4.3 什么时候用

```
简单直接的问题（"鸡蛋多少卡"）→ 不需要改写
模糊的问题（"吃什么好"）→ 需要改写
多轮对话中的指代（"那个呢"）→ 需要结合历史改写
```

## 小结

四种优化手段的优先级：

```
1. 好的分块策略（成本最低，效果最大）
   → 递归分块，在自然边界切分

2. 查询改写（成本低，对模糊问题效果好）
   → 一次 LLM 调用

3. Reranker 重排序（成本中等，提升检索精度）
   → 多一次 LLM 调用

4. 混合检索（成本较高，需要额外的关键词索引）
   → 需要 BM25 支持
```

**不是每个项目都需要全部用上。** 先把分块做好，检索质量不够再逐步加优化。
