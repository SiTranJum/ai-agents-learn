## 1. 产品概述

### 1.1 产品定位

一款 **AI-native** 的个人健康管理产品。AI 不是独立的"聊天机器人"，而是融入产品各个环节的智能助手，通过轻量级交互方式自然地辅助用户管理健康。

**核心价值主张：** 像使用现代 Web 应用一样管理健康，AI 在需要时自然出现，不打断用户流程，让健康管理变得轻松自然。

**产品理念：** AI-native，轻交互融入产品，AI 和传统 UI 无缝结合

### 1.2 产品形态与目标用户

**产品形态：** 移动端 App（iOS/Android），V1 不支持桌面端和平板端

**目标用户：**
- 有健康管理意识的普通用户
- 需要控制饮食/体重的人群
- 健身爱好者
- 慢性病（如痛风、糖尿病）需要关注饮食的人群

**暂不支持：** 智能手环/手表对接、社交社区功能

## 2. 核心设计理念：AI-native 轻交互

### 2.1 什么是 AI-native 轻交互？

**传统 AI 产品：** 有一个独立的"AI 聊天界面"，用户需要主动进入对话页面才能使用 AI。

**AI-native 轻交互：** AI 融入产品的每个角落，通过轻量级的方式出现：
- 底部常驻 AI 输入框，支持文字和语音输入
- 单轮交互以浮层卡片展示结果
- 多轮对话自动切换全屏对话页
- 在数据分析页面生成洞察报告

### 2.2 AI 交互模式设计

**核心设计：底部常驻 AI 输入框 + 混合模式**

#### AI 输入框（全局常驻入口）

**定位：** 全局 AI 助手入口，常驻底部，用户随时可与 AI 交互

**布局结构：**
```
┌─────────────────────────────┐
│         页面内容区           │
├─────────────────────────────┤
│ [📷] [🎤] 说点什么...  [发送] │  ← 常驻 AI 输入框
├─────────────────────────────┤
│  🏠    🍽️    📊    👤       │  ← 底部 Tab 导航
└─────────────────────────────┘
```

**交互方式：**
- 底部常驻输入框，支持文字输入、语音输入、拍照上传
- 用户输入自然语言指令，AI 解析后执行操作
- 单轮交互：结果以浮层卡片展示在输入框上方，操作完成后卡片消失
- 多轮对话：超过 2 轮时自动切换为全屏对话页，完成后返回原页面

**典型场景：**
- "记录今天中午吃了一碗兰州拉面加个蛋"
- "分析一下我最近的健康状况"
- "我的减脂进展怎么样？"
- "帮我制定一个减脂计划"

#### 底部 Tab 导航

**导航结构：**
- 🏠 首页：今日概览 Dashboard
- 🍽️ 饮食：饮食记录与管理
- 📊 数据：身体数据 + 数据分析 + 计划
- 👤 我的：个人档案与设置

**特点：**
- 4 个 Tab，AI 输入框常驻不占 Tab 位
- 当前页面 Tab 高亮
- 点击 Tab 切换页面

## 3. 用户个人信息管理

### 3.1 信息收集策略

**核心原则：** 必填和推荐信息用传统表单（准确性优先）,偏好信息用 AI 渐进收集（自然性优先）

### 3.2 信息分层

| 层级 | 信息项 | 收集方式 | 时机 |
|------|--------|----------|------|
| 必填 | 性别、年龄、身高、体重 | 表单 | 注册/首次使用 |
| 必填 | 健康目标（减脂/增肌/维持/改善饮食） | 表单 | 注册/首次使用 |
| 推荐 | 活动量等级 | 表单 | 注册/首次使用 |
| 推荐 | 饮食偏好/限制（素食、过敏、忌口） | 表单 | 注册/首次使用 |
| 推荐 | 基础疾病（糖尿病、高血压等） | 表单 | 注册/首次使用 |
| 渐进 | 作息习惯（几点吃饭、几点睡觉） | AI 对话 | 使用过程中 |
| 渐进 | 口味偏好（喜欢什么菜系） | AI 对话 | 使用过程中 |
| 渐进 | 运动习惯 | AI 对话 | 使用过程中 |

### 3.3 注册流程设计

**第一步：基础信息（必填）**
- 性别、年龄、身高、体重
- 健康目标（单选：减脂/增肌/维持/改善饮食）

**第二步：交互偏好（必填,与对话模式结合）**
- 选择 AI 交互模式：效率模式/确认模式/学习模式
- 简短说明每种模式的特点

**第三步：补充信息（推荐填写,可跳过）**
- 活动量等级
- 饮食偏好/限制
- 基础疾病

**第四步：完成注册,进入产品**

## 4. AI 交互模式配置

### 4.1 三种交互模式

| 模式 | 信任度 | 信息量 | 行为描述 | 适用场景 |
|------|--------|--------|----------|----------|
| 效率模式 | 高信任 | 简洁 | AI 直接执行,Toast 提示结果,不需要确认 | 老用户、赶时间 |
| 确认模式 | 低信任 | 简洁 | AI 解析后展示结果,用户确认才生效,不啰嗦 | 谨慎型用户 |
| 学习模式 | 低信任 | 详细 | AI 解析后展示结果 + 知识讲解,用户确认才生效 | 想了解营养知识的用户 |

### 4.2 模式影响的交互场景

**场景：饮食记录**

**效率模式：**
```
用户："中午吃了一碗兰州拉面加个蛋"
  ↓
AI 直接记录,Toast 提示："已记录午餐,共 670kcal"
```

**确认模式：**
```
用户："中午吃了一碗兰州拉面加个蛋"
  ↓
AI 展示解析结果：
  - 兰州拉面 1碗 约 600kcal
  - 鸡蛋 1个 约 70kcal
  [确认记录] [修改]
```

**学习模式：**
```
用户："中午吃了一碗兰州拉面加个蛋"
  ↓
AI 展示解析结果 + 知识：
  - 兰州拉面 1碗 约 600kcal
    （主要来自面条的碳水化合物和汤底的油脂）
  - 鸡蛋 1个 约 70kcal
    （优质蛋白质来源,建议每天 1-2 个）
  [确认记录] [修改]
```

## 5. 核心功能设计

### 5.1 饮食记录与管理

#### 5.1.1 每日饮食记录

**核心能力：**
- 记录早/中/晚餐及加餐的摄入情况
- 主要通过 AI 对话录入（Command Bar 或智能输入框）
- 也支持传统表单手动录入（兜底方案）
- 按餐次组织,每餐可添加多个食物条目
- 支持编辑、删除已记录的食物
- 每条记录包含：食物名称、份量、热量、三大营养素、记录时间

**展示方式：**
- 时间轴视图：按时间顺序展示当日饮食
- 餐次视图：按早/中/晚/加餐分组展示
- 支持查看历史日期的饮食记录

#### 5.1.2 自然语言饮食记录

**交互流程：**
1. 用户通过 Command Bar 或饮食记录输入框输入自然语言
2. AI 解析食物名称、份量、热量
3. 根据交互模式展示结果（直接记录/确认/详细讲解）
4. 写入数据库

**AI 解析能力：**
- 识别食物名称（支持俗称、方言）
- 估算份量（"一碗"、"一盘"、"一份"）
- 计算热量和营养成分
- 处理复合食物（"兰州拉面加个蛋"）

#### 5.1.3 食物营养查询

**功能说明：**
- 查询食物的热量、三大营养素（碳水/蛋白质/脂肪）、微量元素、嘌呤等
- 支持文字搜索、语音搜索、AI 对话查询
- 本地数据库优先,查不到时调用第三方 API

**数据源策略：**
- 本地库：手动录入 200-300 种高频中国食物
- 第三方 API：FatSecret / Open Food Facts
- AI 兜底：查不到的食物,LLM 实时估算（标记"AI 估算",允许用户修正）

#### 5.1.4 营养占比展示

**功能说明：**
- 可视化展示当日各营养素占比
- 图表形式：饼图/环形图展示碳水、蛋白质、脂肪比例
- 对比展示：实际摄入 vs 目标值
- 微量元素展示：维生素、矿物质等（P1）

**AI 增强：**
- 图表下方展示 AI 生成的营养点评（如"今日碳水偏高,蛋白质略低"）
- 仅在用户主动查看时生成,不主动消耗 token

#### 5.1.5 热量与营养目标设定

**功能说明：**
- 设定每日热量目标值（基于 TDEE 自动计算推荐值）
- 设定三大营养素目标（碳水/蛋白质/脂肪的克数或比例）
- 实时对比实际摄入 vs 目标值,展示差距
- 支持手动修改目标值

**AI 增强：**
- 首次设定时,AI 基于用户画像（身高、体重、目标、活动量）推荐目标值
- 目标调整时,AI 给出合理性评估（如"1000kcal/天 过低,不利于健康减脂"）

#### 5.1.6 拍照识别

**分层精度策略：**

| 食物类型 | 识别精度 | 处理方式 |
|---------|---------|---------|
| 标准化食物（包装食品、连锁餐厅） | 高（误差 ±5%） | 直接给出结果 |
| 非标准化食物（家常菜、不规则食物） | 中（误差 ±20%） | 根据交互模式决定是否追问 |

**交互流程（以确认模式为例）：**
```
用户拍照上传
  ↓
AI 识别：兰州拉面（非标准化食物）
  ↓
展示结果："兰州拉面 1碗,约 600kcal（误差 ±20%）"
  [直接记录] [提高精度]
  ↓
用户点击"提高精度"
  ↓
展开表单：
  - 碗的大小：[小碗] [中碗] [大碗]
  - 汤底：[清汤] [红汤]
  - 加料：[牛肉] [鸡蛋] [其他]
  ↓
用户补充信息后,AI 重新计算：
  "兰州拉面 1碗（中碗,红汤,加蛋）,约 650kcal（误差 ±10%）"
```

#### 5.1.7 条形码扫描

**实现方案：**
- 移动端：实时扫描模式（打开摄像头,实时识别）
- Web 端：拍照上传模式（上传包装照片,后端识别）

**技术栈：**
- 前端识别：QuaggaJS / ZXing.js
- 后端查询：第三方食物数据库 API（FatSecret / Open Food Facts）
- 降级方案：查不到条形码时,引导用户拍照识别或手动输入

**AI 介入点：**
- 扫描成功后,AI 给出个性化建议（如"这个零食糖分较高,建议少吃"）

#### 5.1.8 常用食物收藏

**功能说明：**
- 用户可收藏常吃的食物,快速记录
- 系统自动统计高频食物,展示"最近常吃"
- 收藏列表支持分类（主食、肉类、蔬菜、零食等）
- 一键记录：点击收藏的食物,自动填入默认份量和热量

**AI 增强：**
- AI 根据历史记录自动识别"常吃食物"模式
- 在饮食记录输入框中优先推荐用户的常用食物

#### 5.1.9 饮食管理功能汇总

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 每日饮食记录 | 早/中/晚/加餐记录,主要通过 AI 对话录入 | P0 |
| 食物营养查询 | 热量、三大营养素、微量元素、嘌呤等 | P0 |
| 营养占比展示 | 饼图/环形图展示营养素占比 | P0 |
| 热量目标设定 | 每日热量/营养素目标,实时对比实际摄入 | P0 |
| 自然语言记录 | AI 解析自然语言,结构化录入 | P0 |
| AI 拍照识别 | 拍照识别食物,分层精度 | P1 |
| 条形码扫描 | 扫描包装食品条形码获取营养信息 | P1 |
| 文字/语音搜索 | 通过文字或语音搜索食物 | P1 |
| 常用食物收藏 | 收藏常吃食物,快速记录 | P2 |

### 5.2 身体数据追踪

#### 5.2.1 基础数据记录

| 数据类型 | 说明 | 优先级 |
|---------|------|--------|
| 体重 | 支持智能预测（规则引擎） | P0 |
| 围度 | 腰围、胸围、臀围、大腿围、上臂围 | P0 |
| BMI | 自动计算 | P0 |
| BMR/TDEE | 自动计算基础代谢和每日总消耗 | P0 |
| 体脂率 | 手动输入或公式估算 | P1 |
| 数据可视化 | 折线图展示变化趋势,支持多指标对比 | P0 |
| 目标值设定 | 为各项指标设定目标值,展示差距 | P1 |
| 历史对比 | 选择任意两个时间点进行数据对比 | P2 |

#### 5.2.2 AI 智能分析报告

**触发方式：**
- 用户主动点击"AI 分析报告"入口
- 通过 Command Bar 提问（如"分析一下我最近的健康状况"）

**AI 分析内容：**
1. **数据关联分析**：发现数据之间的关联模式
   - 示例："睡眠不足 6 小时的日子,第二天饮食热量平均高出 300kcal"

2. **趋势解读**：分析趋势并给出洞察
   - 示例："本月体重下降 3kg,腰围减少 4cm,减脂进展良好"

3. **目标进展评估**：分析目标完成情况
   - 示例："按当前速度,3 个月可能只能减 6kg,要调整目标吗？"

4. **个性化建议**：基于全局数据给出建议
   - 示例："建议增加蛋白质摄入（目前平均 60g/天,建议 80g/天）"

**展示形式：**
- 报告页面：结构化卡片展示,可展开/收起
- 对话模式：通过 Command Bar 多轮对话

### 5.3 辅助健康记录

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 睡眠记录 | 入睡/起床时间、睡眠质量评分 | P1 |
| 饮水追踪 | 每日饮水量记录,目标提醒 | P1 |
| 运动记录 | 运动类型、时长、消耗热量估算 | P1 |
| 生理期记录 | 经期追踪、周期预测 | P1 |
| 排便记录 | 频率、形态（布里斯托分类）记录 | P2 |
| 用药/补剂记录 | 记录日常服用的药物和营养补剂 | P2 |

### 5.4 个人健康档案

**功能说明：**
- 结构化的用户健康画像,所有健康数据的统一入口
- 包含"实际数据"和"目标计划"两套数据,支持对比分析

**档案内容：**
- **核心指标**：身高、体重、BMI、BMR、TDEE、体脂率
- **健康数据**：血压、血糖、尿酸等（用户手动录入体检数据）
- **饮食管理**：饮食偏好、过敏/忌口、每日目标
- **运动燃脂**：运动习惯、运动目标、消耗记录
- **计划数据**：AI 或用户制定的阶段性目标和执行计划

### 5.5 AI 记忆系统

#### 5.5.1 设计目标

让 AI 像一个懂用户的健康顾问：
- 记得用户的稳定特征（过敏、忌口、疾病）
- 知道用户近期的状态（减脂期、平台期）
- 理解用户的偏好习惯（不爱吃辣、喜欢自己做饭）
- 基于记忆提供真正个性化的建议,形成竞品难以复制的壁垒

#### 5.5.2 核心思路：从"对话记忆"到"行为记忆"

产品采用轻交互模式,用户不会长时间对话,因此记忆系统从"记住用户说了什么"转变为"记住用户做了什么、偏好什么"。

#### 5.5.3 三层记忆架构

| 层级 | 范围 | 存储方式 | 内容 | 生命周期 |
|------|------|---------|------|---------|
| 短期记忆<br>(Session Memory) | 单次交互上下文 | LLM 上下文窗口 | Command Bar 一次调用、智能输入框一次确认的上下文 | 交互结束即清空 |
| 中期记忆<br>(Behavioral Memory) | 近期行为模式<br>(7-30 天) | 向量数据库 +<br>结构化字段 | · 饮食偏好（经常吃什么、不吃什么）<br>· 记录习惯（拍照/文字、记录时间段）<br>· 交互偏好（详细建议/简短提示）<br>· 近期目标状态（减重进度、热量达标率） | 7 天未出现→降权<br>30 天未出现→归档到长期记忆 |
| 长期记忆<br>(Profile Memory) | 稳定的个人信息 | 结构化数据库<br>(Supabase) | · 基础信息（身高、体重、年龄、性别、健康状况）<br>· 长期目标（减重目标、健康目标）<br>· 饮食限制（过敏、忌口、宗教禁忌）<br>· 历史里程碑（达成的目标、重要转折点） | 永久,用户可编辑 |

#### 5.5.4 记忆提取机制

**提取触发点：**
1. 饮食记录后 → 提取食物偏好、记录习惯
2. AI 建议被采纳/忽略 → 提取交互偏好
3. 用户主动修改 AI 识别结果 → 提取纠正模式（如用户总是把"米饭"改成"糙米饭"）
4. 计划执行情况 → 提取执行习惯（如周末容易超标）

**提取方式：**
- 每次记录后,后台异步分析（不阻塞用户）
- 识别模式：连续 3 次同类行为 → 标记为偏好
- 避免重复记忆,已有记忆则更新

**用户主动标记：**
- 用户可以在"个人档案"中手动编辑、删除、添加记忆

#### 5.5.5 记忆召回机制

**召回时机：**
- 每次交互时（Command Bar 调用、智能输入框确认）
- 每次生成个性化建议时
- 每次做决策推荐时（如推荐午餐）

**召回优先级：**
1. 长期记忆（必定加载,作为 system prompt 的一部分）
2. 与当前意图强相关的中期记忆（向量检索 Top 3）
3. 与当前时间/场景相关的中期记忆（如"用户早餐通常吃 XX"）

#### 5.5.6 记忆驱动的个性化示例

```
用户：Command Bar 输入"中午吃什么好"

AI 调用记忆：
- 长期记忆：胃不好、不吃香菜、在减脂期
- 中期记忆：最近一周吃了 3 次鸡胸肉
- 短期记忆：今天早上吃了 520kcal

AI 回复：
"你一般中午不怎么吃辣（记得你胃不太好）,
而且你在减脂期,今天早上吃了 520kcal,
午餐建议控制在 500-600kcal。

几个选择：
1. 鸡胸肉饭（约 500kcal）——你上周吃了 3 次,最近可能腻了
2. 清蒸鱼 + 糙米饭（约 480kcal）——换换口味
3. 牛肉蔬菜沙拉（约 420kcal）——蛋白质足,热量低"
```

#### 5.5.7 记忆系统的透明性

**用户可见记忆：**
- 在"个人档案"中提供"AI 记忆"页面
- 展示 AI 记住的所有关于用户的信息（长期记忆 + 中期记忆）
- 用户可以编辑、删除、添加记忆
- 提升用户信任,避免"AI 偷偷记录"的感觉

### 5.6 计划制定与追踪

#### 5.6.1 设计目标

让 AI 像私人教练一样帮用户制定和执行健康计划：
- 通过对话协商制定计划（而不是填表）
- 计划包含明确的目标、时间、执行方案
- AI 持续追踪执行情况,动态调整

#### 5.6.2 核心思路：对话式创建 + 轻量化执行

计划制定是重交互场景（需要多轮对话），但计划执行是轻交互场景（日常记录时自动关联）。

#### 5.6.3 计划类型

| 类型 | 说明 | 示例 | 首版支持 |
|------|------|------|----------|
| 减重计划 | 以降低体重/体脂为目标 | 3 个月减 5kg | ✅ |
| 营养调整计划 | 调整营养比例 | 增加蛋白质摄入到 30% | ✅ |
| 习惯养成计划 | 培养健康习惯 | 每天吃早餐、多喝水 | ✅ |
| 增肌计划 | 以增加肌肉量为目标 | 6 个月增肌 3kg | ❌（预留） |
| 运动计划 | 养成运动习惯 | 每周运动 3 次 | ❌（预留） |

#### 5.6.4 计划制定流程

**触发方式：**
- Command Bar 输入"我想减重" / "制定减脂计划"
- 点击首页"制定计划"按钮

**对话式制定流程：**

```
用户：Command Bar 输入"我想开始减脂"
  ↓
AI 识别意图 → 弹出对话窗口（这是少数需要多轮对话的场景）
  ↓
Step 1：意图识别
AI："你想制定减重计划吗？"
用户确认
  ↓
Step 2：目标确认
AI："你的目标是什么？"
用户："3 个月减 10 斤"
AI 计算：每周需减 0.8 斤,每日热量缺口约 400 kcal
  ↓
Step 3：方案生成
AI 基于用户的长期记忆（基础代谢、饮食偏好）生成方案：
"根据你的基础代谢（1800 kcal）和活动量,建议：
 · 每日热量目标：1600 kcal
 · 营养比例：碳水 45%、蛋白质 30%、脂肪 25%
 · 建议调整：减少晚餐主食、增加蔬菜
 要按这个方案开始吗？"
  [确认] [调整] [取消]
  ↓
Step 4：计划保存
用户确认后,计划存储到数据库,状态：进行中
```

**传统表单兜底：**
- 用户也可以通过表单手动设定计划参数
- 适合对方案有明确想法的用户

#### 5.6.5 计划要素

每个计划包含：
- **目标**：明确、可量化（如"减重 5kg"）
- **周期**：开始日期和结束日期
- **执行方案**：每日热量目标、营养素目标
- **关键节点**：里程碑（如"第一个月减 2kg"）
- **AI 跟进策略**：每周进度报告、动态调整触发条件

#### 5.6.6 计划执行追踪（自动关联）

用户日常记录饮食时,系统自动：
1. 计算当日已摄入热量 vs 目标热量
2. 计算营养比例 vs 目标比例
3. 在记录确认页面展示进度条（轻量化提示）

**实时追踪：**
- 首页展示当前计划进度（完成度百分比、剩余天数）
- 每日达成情况（热量是否达标）

**AI 主动反馈：**
- 每日：基于当日数据,判断是否偏离计划（Toast 提示或卡片）
- 每周：生成周报,展示进度和问题
- 异常时：主动提醒（如"连续 3 天热量超标,是否需要调整？"）

#### 5.6.7 计划修改

**触发场景：**
- 用户主动修改（通过 Command Bar 或计划页面）
- AI 发现计划不合理（如连续 7 天超标 → 建议调整目标）
- 阶段性目标达成/未达成

**修改流程：**
- 重新进入对话窗口
- AI 展示当前计划和执行情况
- AI 分析偏差原因,给出调整建议
- 用户调整目标或方案
- 保存新版本（保留历史版本,支持回溯）

#### 5.6.8 计划完成/放弃

- 达成目标 → AI 祝贺,询问是否制定新计划
- 中途放弃 → 用户手动标记"暂停"或"放弃",AI 不再提示

#### 5.6.9 与其他模块的联动

| 关联模块 | 联动方式 |
|---------|---------|
| 健康档案 | 计划数据作为档案的一部分,实际数据 vs 计划数据对比 |
| AI 记忆 | 计划目标和偏好写入中期/长期记忆 |
| 饮食管理 | 每日热量/营养目标来自当前计划 |
| 身体数据 | 体重/围度目标来自当前计划 |
| AI 分析报告 | 报告中分析计划执行情况 |

## 6. Token 消耗管理与变现模式

### 6.1 用户端额度体系

**免费用户：**
- 每天 10 次"AI 解析"额度（饮食记录解析、拍照识别等）
- 每天 3 次"AI 对话"额度（Command Bar 复杂对话）
- 额度用完后提示："今日 AI 解析次数已用完,明天 0 点刷新。[看广告获取 +3 次] [升级会员]"

**付费用户：**
- 基础会员（¥9.9/月）：每天 50 次 AI 解析 + 20 次 AI 对话
- 高级会员（¥29.9/月）：无限次数
- 年费会员：8 折优惠

### 6.2 广告换额度机制

**规则：**
- 看一次激励视频广告（15-30 秒）→ 获得 3 次 AI 解析额度
- 每天最多看 5 次广告（最多获得 15 次额度）

**广告平台：**
- Web 端：百度联盟
- 移动端 App：穿山甲 + 优量汇（聚合 SDK）
- 微信小程序：优量汇

### 6.3 后台成本控制

| 策略 | 说明 | 预期效果 |
|------|------|---------|
| 场景分级 | 核心场景用好模型,边缘场景用小模型 | 降低 30% 成本 |
| Prompt 优化 | 精简 system prompt,减少不必要的上下文 | 降低 20% token 消耗 |
| 结果缓存 | 相同输入缓存结果 | 降低 40% 重复调用 |
| 规则引擎前置 | 能用规则解决的不调用 LLM | 降低 50% 不必要调用 |
| 批量处理 | 多个轻量任务合并成一次 LLM 调用 | 降低 API 调用次数 |

## 7. 版本规划

### 7.1 版本划分原则

#### V1 内测版
**目标：** 功能完整的内测版，让内部用户体验完整产品形态

**原则：**
- 所有非 AI 核心功能尽量首版上线（页面、表单、数据展示、基础交互）

- 不追求极致性能和完美容错，但要保证核心链路可用
- 用户范围：内部测试用户（5-10 人）

**目标：** 从"内部可用"升级到"真实用户稳定可用"

**原则：**
- 基于 V1 反馈优化体验和准确率
- 补齐容错和边界处理
- 加入运营和数据分析能力
- 用户范围：小范围真实用户（50-100 人）

#### V3 增强版
**目标：** 拉开产品差异化，准备规模化

**原则：**
- 更强个性化和主动服务
- 商业化能力
- 更复杂的 AI 能力扩展

### 7.3 版本实现标准说明

#### V1 内测版的"轻量化"标准

对于复杂的 AI 能力，V1 不追求完美，但要能用：

| 交互模式 | 三种模式基础实现，影响核心场景 | 所有细节 100% 适配、极致个性化 |
| AI 建议 | 不胡说、有针对性、能体现记忆 | 营养师级专业建议、长篇科普报告 |

- 商业化相关：广告、会员、Token 管理后台
