# 健康管家 AI Agent - 产品需求文档（PRD）

> 版本：v0.1（初稿）
> 日期：2026-03-23
> 状态：需求梳理阶段，待确认项较多

---

## 1. 产品概述

一款 **AI-first** 的个人健康管理产品。用户主要通过与 AI 对话来管理健康，AI 负责记录、分析、建议和提醒。不是"健康记录工具 + AI 功能"，而是"AI 助手 + 健康数据支撑"。

**核心价值主张：** 像和朋友聊天一样管理健康，AI 懂你的习惯、记得你的目标、主动关心你的进展。

**产品定位：** AI-first，对话式交互为主，传统表单为辅。

---

## 2. 产品形态与目标用户

**产品形态：** Web 端优先，后续扩展 App 端
**目标用户：** 面向大众

- 有健康管理意识的普通用户
- 需要控制饮食/体重的人群
- 健身爱好者
- 慢性病（如痛风、糖尿病）需要关注饮食的人群

**暂不支持：** 智能手环/手表对接、社交社区功能

---

## 3. 核心设计理念：AI-first 交互

### 3.1 传统模式 vs AI-first 模式

| 场景 | 传统健康 App | 本产品（AI-first） |
|------|------------|-------------------|
| 记录饮食 | 点击"记录" → 搜索食物 → 选份量 → 保存 | "中午吃了一碗牛肉面加个鸡蛋" → AI 自动解析、记录、反馈 |
| 查看建议 | 进入"建议"页面浏览 | AI 主动说"你这周蛋白质摄入偏低，晚餐加个鸡胸肉？" |
| 制定计划 | 填写表单设置目标 | "我想两个月减 5 斤" → AI 反问细节 → 生成计划 → 确认后执行 |
| 查看数据 | 打开各种图表页面 | "我最近体重怎么样？" → AI 生成分析 + 可视化 |
| 拍照记录 | 拍照 → 手动选食物 → 确认 | 拍照 → AI 识别并确认 → "看起来是宫保鸡丁饭，大概 650 大卡，要记录吗？" |

```
┌──────────────────────────────────────────────────────────┐
│                     AI Agent 核心能力                      │
├──────────────┬──────────────┬──────────────┬─────────────┤

├──────────────┼──────────────┼──────────────┼─────────────┤
│ 自然语言理解  │ 短期：对话上下文│ 数据趋势分析  │ 写入健康档案 │
│ 意图识别     │ 中期：近期习惯 │ 异常检测      │ 生成计划    │
│ 食物图像识别  │ 长期：稳定特征 │ 目标进度追踪   │ 推送提醒    │
│ 多轮对话     │ 特征提取存储  │ 周/月报告生成  │ 修改目标    │
│ 模糊输入处理  │ 记忆检索召回  │ 个性化建议    │ 数据可视化   │
└──────────────┴──────────────┴──────────────┴─────────────┘
```

### 3.3 AI 交互场景详解

#### 场景 1：自然语言数据录入

```
用户: 早上喝了杯豆浆，吃了两个包子
AI:   已记录早餐 🔔
      · 豆浆（250ml）：80 大卡
      · 肉包子 ×2：440 大卡
      早餐合计：520 大卡，今日已摄入 520/1800 大卡
      蛋白质偏少，午餐建议加点肉类或蛋类。
      [确认] [修改份量] [这不对]
```

**关键设计：**
- AI 解析自然语言 → 匹配食物数据库 → 估算份量 → 计算热量
- 给出即时反馈 + 结合当日目标的建议
- 提供修正入口，不假定 AI 一定对

#### 场景 2：拍照识别 + 上下文建议

```
用户: [拍照：一盘菜 + 米饭]
AI:   我看到的是：
      · 红烧排骨（约 200g）：480 大卡
      · 米饭（约 150g）：170 大卡
      份量估算可能有偏差，你可以调整。

      你今天午餐加上这些的话，已经到 1400 大卡了。
      你目前在减脂期，晚餐建议控制在 400 大卡以内，
      可以吃个鸡胸肉沙拉或者清蒸鱼 + 蔬菜。
      [记录] [调整份量] [不记录]
```

#### 场景 3：主动提醒与关怀

```
AI → 用户（晚 8 点）:
      今天还没记录晚餐，吃了什么呀？
      如果还没吃，现在吃的话建议控制在 400 大卡以内哦。

AI → 用户（下午 3 点）:
      今天才喝了 500ml 水，记得多喝水～
      你的目标是 2000ml，还差 1500ml。

AI → 用户（周一早上）:
      上周总结：
      · 平均每日摄入 1650 大卡（目标 1800），控制得不错
      · 体重从 75.2 降到 74.8，趋势良好
      · 蛋白质摄入偏低（平均每天 45g，建议 70g+）
      这周要不要调整一下饮食计划？
```

#### 场景 4：对话式计划制定

```
用户: 我想开始减脂
AI:   好的！先了解一下你的情况：
      你现在 75kg，BMI 24.2，属于正常偏上。
      你的目标体重是多少？希望多久达到？

用户: 想降到 70kg，三个月吧
AI:   三个月减 5kg 是比较健康的节奏，大概每周减 0.4kg。
      根据你的 TDEE（2200 大卡），建议：
      · 每日摄入：1700-1800 大卡
      · 蛋白质：70g+（防止肌肉流失）
      · 搭配每周 3 次有氧运动（每次 30min+）

      要按这个方案设定你的计划吗？
      [采纳方案] [调整细节] [再想想]

用户: 可以，就这样
AI:   已更新你的健康计划：
      · 每日热量目标：1800 大卡
      · 蛋白质目标：70g
      · 计划周期：2026.03.23 - 2026.06.23
      · 目标体重：70kg
      我会每天帮你盯着摄入，每周给你出进度报告。加油！
```

#### 场景 5：记忆驱动的个性化

```
用户: 中午吃什么好
AI:   你一般中午不怎么吃辣（上次说胃不太好），
      而且你在减脂期，今天早上吃了 520 大卡，
      午餐建议控制在 500-600 大卡。

      几个选择：
      1. 鸡胸肉饭（约 500 大卡）——你上周吃了 3 次，最近可能腻了
      2. 清蒸鱼 + 糙米饭（约 480 大卡）——换换口味
      3. 牛肉蔬菜沙拉（约 420 大卡）——蛋白质足，热量低

      你今天想吃啥风格的？
```

**记忆在这里的作用：**
- 记得用户胃不好 → 不推荐辣的
- 记得用户在减脂期 → 控制热量
- 记得最近常吃鸡胸肉 → 避免重复推荐

### 3.4 AI 记忆系统设计

```
┌──────────────────────────────────────────────────────┐
│                   AI 记忆系统                         │
├─────────────┬─────────────────┬──────────────────────┤
│   短期记忆   │     中期记忆     │       长期记忆        │
├─────────────┼─────────────────┼──────────────────────┤
│ 当前对话上下文│ 近期饮食偏好     │ 基础身体数据          │
│ 本次会话意图  │ 当前阶段目标     │ 过敏/忌口信息         │
│ 未确认的操作  │ 最近运动习惯     │ 慢性病/用药情况       │
│             │ 近期情绪/状态    │ 稳定的口味偏好        │
│             │ 最近睡眠规律     │ 长期健康目标          │
├─────────────┼─────────────────┼──────────────────────┤
│ 存储：对话上下文│ 存储：数据库      │ 存储：数据库           │
│ 生命周期：单次  │ 生命周期：周-月级  │ 生命周期：永久         │
│ 技术：LLM 上下文│ 技术：向量检索    │ 技术：结构化存储+向量   │
└─────────────┴─────────────────┴──────────────────────┘
```

**记忆提取机制：**
- 对话中 AI 自动识别值得记忆的信息（如"我胃不好"、"我不吃香菜"）
- 提取后分类存入对应层级
- 每次对话开始时，根据上下文召回相关记忆注入 Prompt

### 3.5 AI 与传统界面的分工

| 操作 | 主要方式 | 辅助方式 |
|------|---------|---------|
| 数据录入 | AI 对话（自然语言/拍照） | 传统表单（手动微调） |
| 查看趋势 | AI 生成分析 + 图表 | 数据看板（自主浏览） |
| 制定计划 | AI 对话（多轮协商） | 表单（手动编辑计划） |
| 查询信息 | AI 对话 | 搜索（食物查询） |
| 提醒通知 | AI 主动推送 | 浏览器通知 |
| 查看历史 | AI 汇总分析 | 日历/列表视图 |

**原则：AI 做主路径，传统 UI 做兜底和精细操作。**

| 食物营养查询 | 查询食物的热量、三大营养素（碳水/蛋白质/脂肪）、微量元素、嘌呤等 | P0 |
| 营养占比展示 | 可视化展示各营养素占比（饼图/环形图） | P0 |
| 每日饮食记录 | 记录早/中/晚餐及加餐的摄入情况（主要通过 AI 对话录入） | P0 |
| 热量目标设定 | 设定每日热量/营养素目标值，实时对比实际摄入 | P0 |
| AI 拍照识别 | 拍照识别食物并估算热量（辅助输入，非精确测量） | P1 |
| 条形码扫描 | 扫描包装食品条形码获取营养信息 | P1 |

**数据源依赖：** 需要一个可靠的中文食物营养数据库（中国食物成分表 / USDA FoodData Central / 第三方 API）。

### 4.2 身体数据追踪

| 功能 | 说明 | 优先级 |

| 数据可视化 | 折线图展示变化趋势，支持多指标对比 | P0 |
| 自动指标计算 | BMI、BMR（基础代谢）、TDEE（每日总消耗）自动计算 | P0 |
| 体脂率记录 | 支持手动输入或通过公式估算 | P1 |
| 目标值设定 | 为各项指标设定目标值，展示差距 | P1 |
| 历史对比 | 选择任意两个时间点进行数据对比 | P2 |

### 4.3 辅助健康记录

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 睡眠记录 | 入睡/起床时间、睡眠质量评分 | P1 |
| 饮水追踪 | 每日饮水量记录，目标提醒 | P1 |
| 生理期记录 | 经期追踪、周期预测 | P1 |
| 排便记录 | 频率、形态（布里斯托分类）记录 | P2 |
| 用药/补剂记录 | 记录日常服用的药物和营养补剂 | P2 |
| 运动记录 | 运动类型、时长、消耗热量估算 | P1 |

### 4.4 个人健康档案

一个结构化的用户健康画像，包含以下大项：

- **核心指标**：身高、体重、BMI、BMR、TDEE、体脂率
- **健康数据**：血压、血糖、尿酸等（用户手动录入体检数据）
- **饮食管理**：饮食偏好、过敏/忌口、每日目标
- **运动燃脂**：运动习惯、运动目标、消耗记录
- **计划数据**：AI 或用户制定的阶段性目标和执行计划

**实际数据 vs 计划数据：** 档案中同时维护"实际记录"和"目标计划"两套数据，支持对比分析执行情况。

## 5. 竞品差异化分析

### 5.1 市场现状

| 产品 | 定位 | 弱点（我们的机会） |
|------|------|-------------------|
| 薄荷健康 | 饮食记录 + 社区 | 记录操作繁琐，AI 能力弱，靠手动搜索+选择 |
| Keep | 运动为主 + 饮食辅助 | 饮食管理是附属功能，不够深入 |
| MyFitnessPal | 饮食记录 + 条码扫描 | 国内食物数据弱，无 AI 对话能力 |
| 各类体重记录 App | 单一记录工具 | 只记录不分析，无智能建议 |

### 5.2 我们的差异化壁垒

**核心差异：AI 不是锦上添花，是产品底座。**

| 维度 | 传统产品 | 我们 |
|------|---------|------|
| 交互方式 | 表单驱动（点、选、填） | 对话驱动（说、拍、聊） |
| 数据录入 | 用户主动、手动 | AI 辅助，自然语言解析 |
| 健康建议 | 固定模板、千人一面 | 基于记忆的个性化建议 |
| 用户感受 | "我在填表" | "我在和健康顾问聊天" |
| 计划制定 | 用户自己设定参数 | AI 协商式制定，持续跟进调整 |
| 数据分析 | 展示图表，用户自己看 | AI 主动解读趋势，发现问题 |
| 提醒方式 | 固定时间推送 | 基于用户习惯和状态的智能推送 |

### 5.3 AI 深度结合的三个层次

```
第一层：AI 做翻译（基础）
  用户说自然语言 → AI 转成结构化数据
  例："吃了一碗面" → {食物:面条, 份量:1碗, 热量:350}

第二层：AI 做分析（进阶）
  AI 读取用户数据 → 发现规律和问题 → 给出建议
  例：发现用户连续一周蛋白质不足 → 主动提醒并给方案

第三层：AI 做决策伙伴（核心壁垒）
  AI 理解用户的目标、偏好、限制、习惯 → 像私人教练一样陪伴
  例：知道用户不爱跑步、膝盖不好、周末喜欢做饭 →
      推荐游泳+自己做低卡餐，而不是跑步+外卖沙拉
```

**第三层是真正的壁垒，因为它依赖长期积累的用户记忆，竞品抄不走。**

## 6. 可行性分析

### 6.1 技术可行性

| 功能 | 可行性 | 风险点 |
|------|--------|--------|
| 食物营养查询 | ✅ 高 | 数据源质量和覆盖度是关键 |
| 身体数据追踪 | ✅ 高 | 标准 CRUD + 图表，无技术风险 |
| 饮食记录 | ✅ 高 | UX 设计决定用户留存 |
| AI 拍照识别 | ⚠️ 中 | 食物识别可行，但份量估算精度有限 |
| AI 个性化建议 | ✅ 高 | LLM + RAG 成熟方案 |
| 用户习惯记忆 | ⚠️ 中 | 特征提取和记忆管理需要精心设计 |
| 主动提醒 | ⚠️ 中 | 依赖平台推送能力 |
| 条形码扫描 | ✅ 高 | 需要商品条码数据库 |

### 6.2 关键风险

1. **食物数据库冷启动**：没有好的数据源，产品基础就不牢。建议优先调研可用的中文食物数据 API。
2. **AI 识别精度预期管理**：拍照识别热量误差可能在 20-40%，需要在产品层面管理用户预期，提供手动修正入口。
3. **健康建议的法律边界**：AI 给出的建议不能构成医疗建议，需要明确免责声明，严重情况引导就医。
4. **用户记录疲劳**：饮食记录是高频操作，如果录入太麻烦用户会放弃。AI 辅助录入是关键。
5. **平台选择影响**：App 能力最全但开发成本高，小程序触达好但能力受限，Web 最简单但推送弱。

## 7. MVP 分期建议

### Phase 1：AI 对话 + 核心记录（MVP）
- AI 对话界面（核心入口）
- 自然语言饮食记录（AI 解析 → 写入数据库）
- 食物营养查询（文字搜索 + AI 对话查询）
- 每日饮食记录 + 热量目标
- 身体数据记录 + 折线图
- 个人健康档案（基础版）
- AI 基础建议能力（基于当前数据）

### Phase 2：AI 记忆 + 主动能力
- 用户习惯记忆系统（特征提取 + 向量存储）
- 拍照识别食物
- AI 方案制定（对话式协商）
- AI 数据分析与趋势建议
- AI 主动提醒（Web 通知）
- 辅助记录（睡眠、饮水、运动）

### Phase 3：AI 深度个性化 + 完整体验
- AI 周报/月报自动生成
- 记忆驱动的个性化推荐
- 条形码扫描
- 生理期、排便等辅助记录
- 数据导出
- 计划执行追踪与动态调整

### 食物数据源详细方案

**Phase 1（MVP）：**
- 使用 FatSecret API（免费版或基础付费版）
- 手动录入 200-300 种高频中国食物（米饭、面条、常见菜品等）
- 用户查询流程：本地库 → FatSecret API → 提示"未找到"

**Phase 2：**
- 启用 LLM 辅助估算：查不到的食物，LLM 实时估算营养成分
- 标记"AI 估算"，提示可能有误差，允许用户反馈修正
- 高频查询的 AI 估算结果，人工审核后入库

**Phase 3：**
- 购买或合作获取《中国食物成分表》权威数据
- 替换核心数据，提升准确性
- 建立用户贡献 + 审核机制，持续扩充数据库

**备选方案：**
- FatSecret API: https://platform.fatsecret.com/api/
- Nutritionix API: https://www.nutritionix.com/business/api
- 自建爬虫（不推荐，有法律风险）

## 9. 非功能性需求

- **数据隐私**：健康数据属于敏感个人信息，需加密存储，明确隐私政策
- **免责声明**：AI 建议不构成医疗建议，产品内需明确提示
- **数据备份**：用户数据支持导出和备份
- **离线能力**：基础记录功能应支持离线使用（待确认平台后评估）
