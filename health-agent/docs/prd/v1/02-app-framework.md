# V1 - 应用框架

> 模块：应用框架（首页、导航、AI 输入框、交互模式）
> 优先级：P0（第二批开发）
> 依赖：01-用户体系
> 预计工时：2-3 周

---

## 1. 模块目标

搭建移动端产品的基础骨架，包括首页 Dashboard、底部 Tab 导航、常驻 AI 输入框和三种交互模式。

### 1.1 核心目标
- 首页 Dashboard 提供今日概览和快捷入口
- 底部 Tab 导航清晰，用户能快速到达任何功能
- AI 输入框常驻底部，用户随时可与 AI 交互
- 三种交互模式（效率/确认/学习）影响 AI 输出方式
- 页面内卡片直接更新，减少浮层弹窗

### 1.2 V1 标准
- 移动端优先（iOS/Android），不支持桌面端和平板端
- 首页骨架完整，数据区域先用占位符
- AI 输入框基础功能可用（输入 → 意图识别 → 路由 → 执行）
- 支持对话式修改和表单修改两种方式
- 保守的确认机制，避免误操作

## 2. 功能清单

### 2.1 整体布局结构

#### 2.1.1 底部双层布局

```
┌─────────────────────────────┐
│                             │
│         页面内容区           │
│                             │
│                             │
├─────────────────────────────┤
│ [📷] [🎤] 说点什么...  [发送] │  ← AI 输入框（常驻）
├─────────────────────────────┤
│  🏠    🍽️    📊    👤       │  ← 底部 Tab 导航
└─────────────────────────────┘
```

**布局说明：**
- 最底部：4 个 Tab 导航（首页、饮食、数据、我的）
- 导航上方：常驻 AI 输入框
- 输入框上方：页面内容区

**设计理由：**
- AI 输入框常驻，用户随时可输入，培养"有事找 AI"的习惯
- 导航在最底部，符合拇指操作区
- 输入框在导航上方，点击后键盘弹起时导航被遮住（正常行为）

#### 2.1.2 底部 Tab 导航

| Tab | 图标 | 路由 | 说明 |
|-----|------|------|------|
| 首页 | 🏠 | / | 今日概览 Dashboard |
| 饮食 | 🍽️ | /diet | 饮食记录与管理 |
| 数据 | 📊 | /data | 身体数据 + 数据分析 + 计划 |
| 我的 | 👤 | /profile | 个人档案与设置 |

**Tab 行为：**
- 当前页面 Tab 高亮（图标和文字变色）
- 点击 Tab 切换页面
- 切换页面时 AI 输入框保持常驻

### 2.2 AI 输入框

#### 2.2.1 输入框组成

```
┌─────────────────────────────┐
│ [📷] [🎤] 说点什么...  [发送] │
└─────────────────────────────┘
```

**元素说明：**
- 📷 拍照按钮：点击打开相机或相册，上传食物照片
- 🎤 语音按钮：点击开始语音输入，再次点击结束
- 文字输入框：点击弹起键盘，输入文字
- 发送按钮：点击发送输入内容

**输入框状态：**
- 默认状态：显示 placeholder "说点什么..."
- 输入中：placeholder 消失，显示输入内容
- 有内容时：发送按钮高亮可点击
- 无内容时：发送按钮灰色不可点击

#### 2.2.2 输入处理流程

```
用户输入（文字/语音/拍照）
    ↓
[前端] 发送到后端
    ↓
[后端] LLM 意图识别（结构化输出）
    ↓
返回意图 + 置信度 + 是否需要确认
    ↓
[前端] 判断是否需要确认
    ↓
需要确认 → 弹出确认对话框 → 用户确认后执行
不需要确认 → 直接执行
    ↓
根据意图类型路由到对应处理逻辑
    ↓
执行操作 → 更新页面
```

#### 2.2.3 意图类型

| 意图类型 | 说明 | 示例 | 路由目标 |
|---------|------|------|---------|
| record_diet | 记录饮食 | "今天中午吃了一碗兰州拉面" | 饮食记录模块 |
| modify_diet | 修改饮食记录 | "把午餐的鸡胸肉改成150g" | 饮食记录模块 |
| record_weight | 记录体重 | "今天体重 74.5kg" | 身体数据模块 |
| query_nutrition | 查询营养 | "鸡胸肉的热量是多少" | 营养查询 |
| query_analysis | 查询分析 | "我最近一周的热量趋势" | 数据分析模块 |
| create_plan | 创建计划 | "帮我制定一个减脂计划" | 计划系统 |
| query_plan | 查询计划 | "我的减脂进展怎么样" | 计划系统 |
| general_advice | 健康建议 | "我今天应该吃什么" | AI 建议系统 |
| preference | 用户偏好 | "我喜欢吃鸡胸肉" | AI 记忆系统（后台） |

### 2.3 页面内卡片交互模式

#### 2.3.1 核心设计理念

**不使用浮层弹窗，直接在页面卡片上操作：**
- AI 解析结果直接更新对应的页面卡片
- 卡片上显示确认/修改/取消按钮
- 用户在卡片上完成确认或修改操作

#### 2.3.2 饮食记录场景示例

```
饮食页面：
┌─────────────────────────────┐
│ 早餐 (已记录)                │
│ ┌─────────────────────────┐ │
│ │ 🍞 全麦面包  120kcal    │ │
│ │ 🥛 牛奶      150kcal    │ │
│ └─────────────────────────┘ │
│                             │
│ 午餐 (待确认)  ← AI 刚解析的 │
│ ┌─────────────────────────┐ │
│ │ 🍗 鸡胸肉    165kcal    │ │
│ │                         │ │
│ │ [✓ 确认] [✏️ 修改] [✗ 取消] │ │  ← 操作按钮
│ └─────────────────────────┘ │
│                             │
│ 晚餐 (未记录)                │
│ ┌─────────────────────────┐ │
│ │ + 点击记录晚餐           │ │
│ └─────────────────────────┘ │
├─────────────────────────────┤
│ [📷] [🎤] 说点什么...  [发送] │
├─────────────────────────────┤
│  🏠    🍽️    📊    👤       │
└─────────────────────────────┘
```

**交互流程：**
1. 用户在输入框输入："午餐吃了鸡胸肉"
2. AI 解析 → 识别为午餐记录
3. 午餐卡片状态变为"待确认"
4. 卡片内容更新：显示"鸡胸肉 165kcal"
5. 卡片底部出现：[✓ 确认] [✏️ 修改] [✗ 取消]
6. 用户点击确认 → 保存到数据库，卡片状态变为"已记录"

#### 2.3.3 卡片状态

| 状态 | 说明 | 显示 |
|------|------|------|
| empty | 未记录 | "+ 点击记录" 提示 |
| pending | 待确认 | 显示内容 + 操作按钮 |
| recorded | 已记录 | 显示内容，无操作按钮 |
| editing | 编辑中 | 跳转到详情页 |

### 2.4 修改记录的两种方式

#### 2.4.1 方式 1：详情页修改（表单式）

**适用场景：** 大幅修改、改多个字段、不确定怎么说

**流程：**
```
用户点击卡片上的 [✏️ 修改]
    ↓
跳转到详情页 /diet/record/{id}/edit
    ↓
表单预填充当前记录：
  - 食物名称：[鸡胸肉]
  - 份量：[100] g
  - 餐次：[午餐]
    ↓
用户修改后点 [保存]
    ↓
返回饮食页面，卡片更新
```

#### 2.4.2 方式 2：对话式修改（AI 理解）

**适用场景：** 小幅修改、只改一个字段、用户习惯对话

**流程：**
```
用户在输入框说："把午餐的鸡胸肉改成150g"
    ↓
AI 解析意图：
{
  "intent": "modify_diet",
  "target": {"meal": "lunch", "food": "鸡胸肉"},
  "change": {"amount": 150, "unit": "g"}
}
    ↓
前端定位到午餐卡片中的"鸡胸肉"记录
    ↓
更新卡片内容：鸡胸肉 150g 248kcal
    ↓
卡片重新显示 [✓ 确认] [✏️ 修改] [✗ 取消]
```

**处理歧义：**
- 如果午餐有多条鸡胸肉 → AI 追问："午餐有两条鸡胸肉记录，你要改哪一条？"
- 如果无法定位 → 提示："我没找到这条记录，你可以点击修改按钮进入详情页"

### 2.5 跨页面操作处理

#### 2.5.1 操作规则

| 场景 | 当前页面 | AI 识别意图 | 处理方式 |
|------|---------|------------|---------|
| 记录饮食 | 饮食页面 | record_diet | 直接更新对应餐次卡片 |
| 记录饮食 | 其他页面 | record_diet | Toast 提示 + 自动跳转到饮食页面 + 更新卡片 |
| 记录体重 | 数据页面 | record_weight | 直接更新体重卡片 |
| 记录体重 | 其他页面 | record_weight | Toast 提示 + 自动跳转到数据页面 + 更新卡片 |
| 创建计划 | 任何页面 | create_plan | 确认对话框 + 跳转到数据页面 + 全屏对话 |
| 查询分析 | 任何页面 | query_analysis | 确认对话框 + 跳转到数据页面 + 展示结果 |
| 营养查询 | 任何页面 | query_nutrition | 浮层卡片展示结果（不跳转） |
| 健康建议 | 任何页面 | general_advice | 浮层卡片展示结果（不跳转） |

**规则总结：**
- **页面内操作**（如在饮食页面记录饮食）→ 直接更新页面卡片
- **跨页面操作**（如在饮食页面创建计划）→ 确认对话框 + 自动跳转
- **通用查询**（如营养查询、建议）→ 浮层卡片，不跳转

#### 2.5.2 跨页面操作示例

```
用户在饮食页面输入："帮我制定减脂计划"
    ↓
AI 识别为 create_plan（需要跳转）
    ↓
弹出确认对话框：
  "检测到你想创建计划，需要跳转到计划页面"
  [去创建] [取消]
    ↓
用户点击 [去创建]
    ↓
跳转到数据 Tab
    ↓
进入全屏对话页，开始多轮对话创建计划
```

### 2.6 误操作风险的保守机制

#### 2.6.1 意图置信度阈值

| 操作类型 | 置信度要求 | 确认机制 |
|---------|-----------|---------|
| 页面内记录（如记录饮食） | > 0.7 | 卡片显示确认按钮 |
| 页面内查询（如营养查询） | > 0.6 | 直接展示结果 |
| 跨页面操作（如创建计划） | > 0.8 | 弹出确认对话框 |
| 修改记录 | > 0.7 | 卡片重新显示确认按钮 |
| 删除记录 | > 0.9 | 二次确认对话框 |

**低置信度处理：**
```
if 置信度 < 阈值:
    弹出确认对话框：
    "你是想{intent_description}吗？"
    [是的] [不是]
```

#### 2.6.2 关键操作二次确认

**删除记录：**
```
用户说："删除午餐记录"
    ↓
AI 识别为删除操作
    ↓
弹出确认对话框：
  "确定删除午餐记录吗？删除后无法恢复"
  [确定删除] [取消]
```

**跨页面操作：**
```
用户在饮食页面说："我想减肥"
    ↓
AI 识别为 create_plan（置信度 0.75，低于 0.8）
    ↓
弹出确认对话框：
  "你是想创建减脂计划吗？"
  [是的] [不是]
```

### 2.8 首页 Dashboard

#### 2.8.1 页面布局

首页从上到下分为以下区域：

**区域 1：今日概览卡片（顶部）**
- 今日已摄入热量 / 目标热量
- 进度条展示
- 三大营养素（碳水/蛋白质/脂肪）占比

**区域 2：快捷操作（中部）**
- 快速记录饮食（点击后输入框获得焦点）
- 快速记录体重
- 查看今日计划

**区域 3：今日饮食时间轴（中部）**
- 按餐次展示今日已记录的饮食
- 每条记录显示：食物名称、热量、时间
- 未记录的餐次显示"未记录"提示

**区域 4：AI 洞察卡片（底部）**
- 一句话 AI 洞察（如"今天蛋白质摄入偏低，建议晚餐补充"）
- 点击可展开详细建议

**区域 5：计划进度卡片（底部）**
- 当前活跃计划的进度
- 今日任务完成情况

#### 2.8.2 数据刷新
- 页面加载时获取最新数据
- 记录饮食后自动刷新首页数据
- 下拉刷新

### 2.9 交互模式

#### 2.9.1 三种模式定义

**效率模式：**
- 目标：最快完成操作
- AI 行为：直接执行，不多问
- 适合：熟练用户、重复操作
- 示例：用户说"记录100g鸡胸肉" → AI 直接记录，Toast 提示"已记录"

**确认模式（默认）：**
- 目标：准确性和用户掌控感
- AI 行为：展示解析结果，等用户确认后执行
- 适合：大多数用户
- 示例：用户说"记录100g鸡胸肉" → AI 更新卡片 → 用户点击"确认记录"

**学习模式：**
- 目标：帮助用户学习健康知识
- AI 行为：展示结果 + 附带知识讲解
- 适合：新手用户、想学习的用户
- 示例：用户说"记录100g鸡胸肉" → AI 更新卡片 + "鸡胸肉是优质蛋白质来源..."

#### 2.9.2 模式切换

**设置入口：**
- 注册完成后选择（推荐确认模式）
- 个人中心 → 设置 → 交互模式

**切换方式：**
- 设置页面单选切换
- 立即生效

#### 2.9.3 模式影响范围（V1）

| 场景 | 效率模式 | 确认模式 | 学习模式 |
|------|---------|---------|---------|
| 饮食记录 | 直接记录 + Toast | 卡片显示确认按钮 | 卡片 + 确认 + 知识 |
| 营养查询 | 直接展示结果 | 展示结果 | 结果 + 详细讲解 |
| AI 建议 | 简短建议 | 标准建议 | 详细建议 + 原因 |
| 计划创建 | 快速创建（减少追问） | 标准对话创建 | 详细对话 + 知识讲解 |

## 3. 页面设计要点

### 3.1 AI 输入框
- 高度：48px（符合移动端点击区域标准）
- 背景色：白色或浅灰色，与导航区分
- 圆角：8px
- 阴影：轻微上阴影，突出层级
- 语音按钮：长按录音，松开发送（或点击开始/结束）

### 3.2 页面卡片
- 卡片圆角：12px
- 卡片间距：12px
- 卡片阴影：轻微阴影
- 待确认状态：边框高亮或背景色变化
- 操作按钮：清晰可见，间距合理

### 3.3 确认对话框
- 半透明背景遮罩
- 对话框居中显示
- 圆角：16px
- 按钮清晰：主操作按钮高亮，次要操作按钮灰色

### 3.4 全屏对话页
- 顶部导航栏：返回按钮 + 标题
- 对话区域：白色背景，用户消息右对齐，AI 消息左对齐
- 输入框：固定在底部，样式与常驻输入框一致

### 3.5 首页 Dashboard
- 卡片式布局，每个区域独立卡片
- 卡片间距：12px
- 卡片圆角：8px
- 卡片阴影：轻微阴影

## 4. 交互细节

### 4.1 键盘弹起行为
- 点击输入框后，键盘从底部弹起
- 输入框随键盘上移，保持在键盘上方
- 底部 Tab 导航被键盘遮住（正常行为）
- 页面内容区域可滚动

### 4.2 语音输入
- 点击语音按钮开始录音，按钮变红
- 再次点击结束录音，自动发送
- 录音过程中显示音量波形动画
- 支持取消：录音中向上滑动取消

### 4.3 拍照上传
- 点击拍照按钮，弹出选择：拍照 / 从相册选择
- 选择照片后，显示预览
- 确认后上传，显示上传进度
- 上传完成后 AI 识别，更新对应卡片

### 4.4 错误处理
- 意图识别失败：显示"抱歉，我没理解你的意思，可以换个说法试试"
- 网络错误：显示"网络异常，请稍后重试"
- LLM 超时：显示"AI 正在思考中..."（超过 10 秒显示超时提示）
- 记录定位失败：提示"未找到记录，请点击修改按钮"

## 5. 测试要点

### 5.1 AI 输入框测试
- 文字输入、语音输入、拍照上传
- 各种意图识别准确性
- 多意图识别
- 模糊输入处理
- 错误输入处理
- 响应时间 < 3 秒

### 5.2 页面内卡片交互测试
- 卡片状态切换（empty → pending → recorded）
- 确认/修改/取消按钮功能
- 卡片内容更新

### 5.3 修改记录测试
- 详情页修改：跳转、预填充、保存、返回
- 对话式修改：简单场景、复杂场景、歧义处理
- 修改后卡片更新

### 5.4 跨页面操作测试
- 自动跳转功能
- 确认对话框
- Toast 提示
- 跳转后页面状态

### 5.5 误操作防护测试
- 意图置信度判断
- 低置信度确认机制
- 关键操作二次确认
- 删除操作确认

### 5.6 交互模式测试
- 三种模式切换
- 各模式下饮食记录流程
- 各模式下建议输出差异

### 5.7 首页测试
- 数据加载和展示
- 空状态展示（新用户无数据）
- 数据刷新
- 下拉刷新

### 5.8 导航测试
- Tab 切换
- 页面跳转
- 返回行为
