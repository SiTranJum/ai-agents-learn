# 健康管家 Agent - 产品设计方案总结

## 文档说明

本文档是基于头脑风暴会议（`master-prd-redesign-brainstorming.md`）的产品设计方案总结，提炼了核心产品理念、关键设计决策和技术方向。

---

## 一、核心产品理念

### 1.1 产品定位

**AI-native 的个人健康管理产品，以轻量级交互为核心，AI 无处不在但不打扰用户。**

### 1.2 核心差异化

| 传统健康 App | 我们的产品 |
|-------------|-----------|
| 独立的 AI 聊天界面 | AI 融入每个输入框，无需专门对话页面 |
| 填表单式的数据记录 | 自然语言 + 拍照 + 扫码，极简记录 |
| 固定的交互模式 | 三种可配置模式（效率/确认/学习） |
| 用户主动查看数据 | AI 主动发现洞察，轻量级卡片呈现 |
| Token 概念暴露给用户 | 用户只看到"次数"，可看广告换额度 |

---

## 二、核心设计决策

### 2.1 AI 交互模式

**方案：Command Bar + 智能输入框增强**

**Command Bar（顶部全局搜索栏）**
- 用户主动唤醒 AI 的入口（点击或快捷键 Ctrl+K）
- 处理复杂的自然语言交互和多轮对话
- 类似 Notion Quick Find 或 Linear Command K

**智能输入框增强**
- 在页面原有输入框中直接集成 AI 能力
- 后端智能路由：LLM / 规则引擎 / 数据库查询
- 示例：
  - 体重输入框 → 下方出现 AI 预测选项（规则引擎）
  - 饮食记录输入框 → 实时解析自然语言（LLM）
  - 测量指导 → 点击问号图标展开帮助文案（静态）

**LLM vs 规则引擎路由规则**

| 场景 | 路由目标 | 原因 |
|------|----------|------|
| 自然语言解析 | LLM | 非结构化文本理解 |
| 模糊输入追问 | LLM | 需要上下文和对话能力 |
| 个性化建议生成 | LLM | 综合多维度信息 |
| 体重预测 | 规则引擎 | 数学公式即可 |
| 历史数据推荐 | 数据库查询 | 简单查询 |
| 测量指导 | 预设文案 | 固定内容 |

---

### 2.2 用户个人信息管理

**信息收集策略：表单 + AI 渐进收集**

| 层级 | 信息项 | 收集方式 | 时机 |
|------|--------|----------|------|
| 必填 | 性别、年龄、身高、体重、健康目标 | 表单 | 注册/首次使用 |
| 推荐 | 活动量等级、饮食偏好/限制、基础疾病 | 表单（可跳过） | 注册/首次使用 |
| 渐进 | 作息习惯、口味偏好、运动习惯 | AI 对话自然获取 | 使用过程中 |

**设计原则：**
- 必填和推荐项用传统表单（信息必须准确，AI 对话存在不准确风险）
- 渐进收集用 AI 对话（自然、不打扰）

---

### 2.3 对话风格/模式可配置

**三种交互模式：**

| 模式 | 信任度 | 信息量 | 行为描述 | 适用场景 |
|------|--------|--------|----------|----------|
| 效率模式 | 高信任 | 简洁 | AI 直接执行，Toast 提示结果，不需要确认 | 老用户、赶时间 |
| 确认模式 | 低信任 | 简洁 | AI 解析后展示结果，用户确认才生效，不啰嗦 | 谨慎型用户 |
| 学习模式 | 低信任 | 详细 | AI 解析后展示结果 + 知识讲解，用户确认才生效 | 想了解营养知识的用户 |

**实现方式：**
- 注册时在表单中增加"交互偏好"选项
- 用户后续可以在设置中随时切换
- 模式影响所有 AI 交互的行为

---

### 2.4 拍照识别精度优化

**方案：分层精度 + 智能追问**

**交互流程：**
1. 用户拍照上传
2. AI 识别并判断食物类型
   - 标准化食物（包装食品、连锁餐厅）→ 直接给出结果
   - 非标准化食物（家常菜、不规则食物）→ 根据交互模式决定行为

**不同模式下的行为：**

| 交互模式 | 行为 |
|---------|------|
| 效率模式 | 直接记录，不追问，接受误差。显示："兰州拉面 1碗，约 600kcal（误差 ±20%）" |
| 确认模式 | 给出结果 + "提高精度"选项。用户可选择："直接记录" 或 "提高精度"（展开表单：重量/份量/做法） |
| 学习模式 | 主动追问 + 解释原因。"拉面的热量主要取决于面条重量和汤底油脂，告诉我碗的大小可以更准确估算。大概多大？（小碗/中碗/大碗）" |

---

### 2.5 条形码扫描

**技术方案：混合模式**
- 移动端：实时扫描（类似支付宝扫一扫）
- Web 端：拍照上传模式
- 降级方案：条形码识别失败或数据库查不到 → 拍照识别或手动输入

**数据库策略：**
- 初期：依赖第三方 API（薄荷健康、FatSecret、Open Food Facts）
- 后期：自建数据库 + 用户贡献（众包模式）

**AI 的作用：**
- 条形码识别本身不需要 LLM（用传统 CV 算法）
- AI 只在扫描成功后介入：给出个性化建议（如"这个零食糖分较高，建议少吃"）

### 2.7 Token 消耗管理与付费

**核心原则：用户不需要知道 token 的概念**

**用户端额度体系：**

**免费用户：**
- 每天 10 次"AI 解析"额度（饮食记录解析、拍照识别等）
- 每天 3 次"AI 对话"额度（Command Bar 复杂对话）
- 额度用完后提示："今日 AI 解析次数已用完，明天 0 点刷新。想继续使用？[看广告获取 +3 次] [升级会员]"

**广告换额度机制：**
- 看一次激励视频广告（15-30 秒）→ 获得 3 次 AI 解析额度
- 每天最多看 5 次广告（最多获得 15 次额度）

**付费用户：**
- 基础会员（¥9.9/月）：每天 50 次 AI 解析 + 20 次 AI 对话
- 高级会员（¥29.9/月）：无限次数
- 年费会员：8 折优惠

**后台成本控制层（对用户不可见）：**

| 组件 | 功能 |
|------|------|
| Token 计量系统 | 记录每次 LLM 调用的 token 消耗 |
| 成本预算管理 | 设置每个用户的成本上限（免费 ¥2/月，付费 ¥20/月） |
| 智能路由 | 根据成本优先级选择模型（qwen-plus / qwen-turbo） |
| 缓存系统 | 相同问题复用结果，不重复调用 LLM |
| 异常检测 | 识别异常消耗行为，自动限流 |

**成本优化策略：**
- 场景分级：核心场景用好模型，边缘场景用小模型（降低 30% 成本）
- Prompt 优化：精简 system prompt（降低 20% token 消耗）
- 结果缓存：相同输入缓存结果（降低 40% 重复调用）
- 规则引擎前置：能用规则解决的不调用 LLM（降低 50% 不必要调用）
- 批量处理：多个轻量任务合并成一次 LLM 调用

## 三、技术架构要点

### 3.1 前端架构

**核心组件：**
- Command Bar 组件（全局搜索栏）
- 智能输入框组件（统一封装，支持下拉建议、实时解析、帮助提示）
- AI 建议卡片组件（可收起，出现在相关页面）

### 3.2 后端架构

**核心模块：**
- 智能路由层：根据输入类型分发到 LLM / 规则引擎 / 静态数据
- Token 计量系统：记录每次调用的消耗和成本
- 额度管理系统：检查用户额度，扣除额度，重置额度
- 缓存系统：缓存常见问题的结果

### 3.3 LLM 选型

- 学习阶段：国内 LLM（API 便宜），如通义千问（qwen-plus / qwen-turbo）
- 生产阶段：根据成本和效果评估，可能混合使用多个模型

## 四、关键指标

### 4.1 用户体验指标
- 饮食记录平均耗时（目标：< 30 秒）
- AI 解析准确率（目标：> 85%）
- 用户对 AI 建议的采纳率

### 4.2 成本指标
- 每日平均 token 消耗 / 用户
- 成本 / 收入比（LLM 成本 vs 广告收入 + 会员收入）
- 免费用户转化率（看广告 → 付费）

### 4.3 留存指标
- 7 日留存率
- 30 日留存率
- 额度用尽后的用户流失率
