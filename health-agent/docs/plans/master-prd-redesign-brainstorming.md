# Master PRD 重新设计 - 头脑风暴记录

## 背景

初版 PRD 已完成，现需要重新设计优化。核心理念有部分修改，不推翻现有设计，但需要深度优化。

## 核心问题清单

### 1. AI 交互模式（进行中）

**用户需求：**
- 不是传统的"AI 聊天界面"
- 不是固定的悬浮按钮/tab
- 希望 AI **完全隐形**，页面看不到 AI 助手存在
- 用户可以主动唤醒 AI
- AI 可以主动出现（感知用户操作）
- AI 能"看到"当前页面上下文
- AI 可以直接操作页面元素，操作后弹出确认框
- 对话窗口太重，内嵌又不太合适

**用户决策：**
- 采用 **D2（顶部 Command Bar）+ 智能输入框增强** 组合方案
- **Command Bar**：通用 LLM 入口，处理复杂的自然语言交互
- **智能输入框增强**：在页面原有的输入框中直接集成 AI 能力（而不是额外的浮层）
  - 后端根据场景自动路由到 LLM 或规则引擎

**具体交互示例：**

1. **体重输入框**
   - 用户点击体重输入框
   - 输入框下方自动出现 AI 预测的体重选项（基于规则引擎计算）
   - 例如："78.5kg（基于本周热量预测）" "79.0kg（上次体重）"
   - 用户可以直接点击选择，或手动输入

2. **饮食记录输入框**
   - 用户在输入框输入："中午吃了一碗兰州拉面加个蛋"
   - AI（LLM）实时解析，输入框下方出现结构化结果：
     - 兰州拉面 1碗 约 600kcal
     - 鸡蛋 1个 约 70kcal
   - 用户确认或修改后提交

3. **围度测量输入框**
   - 用户不知道怎么测量腰围
   - 输入框旁边有个小问号图标，点击后输入框下方展开简短说明
   - "站直，在肚脐上方 2cm 处水平测量"
   - 这个不需要 LLM，是预设的帮助文案

**LLM vs 规则引擎路由分类：**

| 场景 | 路由目标 | 原因 |
|------|----------|------|
| 自然语言解析（"一碗兰州拉面加个蛋"） | LLM | 非结构化文本理解 |
| 模糊输入追问（"吃了点水果" → 追问） | LLM | 需要上下文理解和对话能力 |
| 个性化建议生成 | LLM | 需要综合多维度信息 |
| 体重预测 | 规则引擎 | 数学公式即可 |
| 历史数据推荐（"你上次填的是 79kg"） | 数据库查询 | 简单查询 |
| 测量指导（"腰围怎么量"） | 预设文案 | 固定内容 |
| 常吃食物推荐 | 统计排序 | 基于历史频率 |
| 数据趋势判断 | 简单统计 | 数学计算 |
- 前端：输入框组件统一封装，支持下拉建议、实时解析、帮助提示
- 后端：智能路由层，根据输入类型分发到 LLM / 规则引擎 / 静态数据

**待探讨方案：**

**隐形模式方案：**
- 方案 C1：语音唤醒 - 用户说"嘿，助手"或按住快捷键说话，AI 通过语音回应并操作页面
  - 优点：完全隐形，自然交互
  - 缺点：需要麦克风权限，公共场合不方便

- 方案 C2：快捷键 + Toast 提示 - 用户按快捷键（如 Ctrl+K），页面顶部出现轻量级输入框，输入后 AI 直接操作页面，操作完成后 Toast 提示
  - 优点：轻量、快速、不打断用户流程
  - 缺点：需要用户记住快捷键

- 方案 C3：AI 主动出现（智能感知）- AI 感知用户行为，页面右下角出现小气泡"需要帮忙记录吗？"，点击后出现轻量级输入框
  - 优点：主动服务，不需要用户唤醒
  - 缺点：可能打扰用户，需要精准的触发时机

- 方案 C4：混合模式 - 结合快捷键唤醒 + AI 主动出现，用户可以在设置中选择是否允许 AI 主动出现

**非隐形模式方案：**

- 方案 D1：固定悬浮球 + 侧边栏对话
  - 页面右下角固定悬浮球（类似客服图标），点击后从右侧滑出对话面板
  - AI 能看到当前页面上下文，对话中可以直接操作页面元素
  - 操作后在页面上高亮显示变化，用户确认后生效
  - 优点：成熟的交互模式，用户熟悉，实现简单
  - 缺点：占用屏幕空间，有一定视觉干扰

- 方案 D2：顶部全局搜索栏（Command Bar）
  - 页面顶部有一个搜索栏（类似 Notion 的 Quick Find 或 Linear 的 Command K）
  - 点击或按快捷键激活，输入自然语言指令
  - AI 解析后直接操作页面，操作完成后收起搜索栏，Toast 提示结果
  - 优点：轻量、不打断流程、符合现代 Web 应用趋势
  - 缺点：不支持多轮对话，适合单次指令

- 方案 D3：页面底部 Dock 栏
  - 页面底部有一个可收起的 Dock 栏（类似 macOS Dock）
  - 默认收起只显示一个小标签，鼠标悬停或点击后展开
  - 展开后显示输入框和最近的对话历史（最多3条）
  - 优点：不占用主要视觉空间，支持简单的多轮对话
  - 缺点：移动端适配困难

- 方案 D4：智能浮层（Contextual Overlay）
  - 不是固定位置的悬浮球，而是根据用户操作动态出现的浮层
  - 例如：用户在饮食记录页面停留3秒，页面中央出现半透明浮层"告诉我你吃了什么"
  - 输入后浮层消失，AI 在后台处理，处理完成后 Toast 提示
  - 优点：主动引导，不需要用户寻找入口
  - 缺点：可能打扰用户，需要精准的触发逻辑

### 2. 用户个人信息管理（已确认）

**用户需求：**
- 需要用户提供个人信息用于和 AI 交互
- 具体需要哪些信息？如何收集？如何使用？

**用户决策：**
- 必填 + 推荐填写 → **传统表单**（信息必须准确，AI 对话存在不准确风险，且这些信息可以快速填写，用 AI 反而太重）
- 渐进收集 → **AI 对话自然获取**

**信息分层：**

| 层级 | 信息项 | 收集方式 | 时机 |
|------|--------|----------|------|
| 必填 | 性别、年龄、身高、体重 | 表单 | 注册/首次使用 |
| 必填 | 健康目标（减脂/增肌/维持/改善饮食） | 表单 | 注册/首次使用 |
| 推荐 | 活动量等级 | 表单 | 注册/首次使用 |
| 推荐 | 饮食偏好/限制（素食、过敏、忌口） | 表单 | 注册/首次使用 |
| 推荐 | 基础疾病（糖尿病、高血压等） | 表单 | 注册/首次使用 |
| 渐进 | 作息习惯（几点吃饭、几点睡觉） | AI 对话 | 使用过程中 |
| 渐进 | 口味偏好（喜欢什么菜系） | AI 对话 | 使用过程中 |
| 渐进 | 运动习惯 | AI 对话 | 使用过程中 |

### 3. 对话风格/模式可配置（已确认）

**用户需求：**
- 支持用户选择对话风格、对话模式等交互细节
- 场景 1：用户嫌 LLM 话痨，只需要少量对话，相信 LLM 所说的内容
- 场景 2：不相信 LLM，需要自己亲手确认
- 场景 3：需要学习相关知识，需要 LLM 提供详细的知识

**用户决策：**
- 抽象为两个维度：**信任度（AI 自主权）** + **信息量（AI 输出详细度）**
- 组合成三种有效模式
- 在注册表单中让用户选择（与第2点的个人信息收集结合）

**模式定义：**

| 模式 | 信任度 | 信息量 | 行为描述 | 适用场景 |
|------|--------|--------|----------|----------|
| 效率模式 | 高信任 | 简洁 | AI 直接执行，Toast 提示结果，不需要确认 | 老用户、赶时间 |
| 确认模式 | 低信任 | 简洁 | AI 解析后展示结果，用户确认才生效，不啰嗦 | 谨慎型用户 |
| 学习模式 | 低信任 | 详细 | AI 解析后展示结果 + 知识讲解，用户确认才生效 | 想了解营养知识的用户 |

**实现方式：**
- 注册时在表单中增加"交互偏好"选项，让用户选择三种模式之一
- 用户后续可以在设置中随时切换
- 模式影响所有 AI 交互的行为（Command Bar、智能输入框增强等）

### 4. 拍照识别精度优化（已确认）

**用户需求：**
- AI 拍照识别热量误差大
- 需要用户提供图片相关信息：食物大概重量、用户已知的某类食物热量等
- 支持用户选择是否需要更精确的热量估算（结合第3点的配置）

**用户决策：**
- 采用 **方案 A（分层精度）+ 方案 B（智能追问）** 的混合方案
- 结合第3点的交互模式，不同模式下行为不同

**交互流程：**

1. **用户拍照上传**
2. **AI 识别并判断食物类型**：
   - 标准化食物（包装食品、连锁餐厅）→ 直接给出结果，误差小
   - 非标准化食物（家常菜、不规则食物）→ 根据交互模式决定行为
3. **根据交互模式响应**：

| 交互模式 | 行为 |
|---------|------|
| 效率模式 | 直接记录，不追问，接受误差。显示："兰州拉面 1碗，约 600kcal（误差 ±20%）" |
| 确认模式 | 给出结果 + "提高精度"选项。用户可选择："直接记录" 或 "提高精度"（展开表单：重量/份量/做法） |
| 学习模式 | 主动追问 + 解释原因。"拉面的热量主要取决于面条重量和汤底油脂，告诉我碗的大小可以更准确估算。大概多大？（小碗/中碗/大碗）" |

4. **用户补充信息后**（如果选择提高精度或回答追问）：
   - AI 重新计算："兰州拉面 1碗（约 500g），约 650kcal（误差 ±10%）"
   - 用户确认后记录

**补充信息类型：**
- 食物重量（克数或份量描述：小碗/中碗/大碗）
- 用户已知的某类食物热量（可选，如果用户说"我知道这个拉面是 XX kcal"）
- 特殊做法（清汤/红汤、加辣油等）

### 5. 条形码扫描（已确认）

**用户需求：**
- 不太懂原理，需要讨论如何实现
- 需要教学具体原理

**条形码扫描原理讲解：**

**1. 什么是条形码？**
- 条形码（Barcode）是一种用黑白条纹编码数字/字母的标识符
- 最常见的是 **EAN-13**（13位数字，国际通用）和 **UPC**（12位数字，北美常用）
- 每个商品的条形码是唯一的，由厂商申请并印在包装上

**2. 扫描识别流程：**
```
用户拍照/扫描
  ↓
前端调用条形码识别库（如 ZXing、QuaggaJS）
  ↓
识别出条形码数字（如：6901028075831）
  ↓
后端拿着这个数字去查询食物数据库
  ↓
返回食物信息（名称、品牌、营养成分、热量等）
```

**3. 技术实现：**

| 环节 | 技术方案 | 说明 |
|------|----------|------|
| 前端识别 | QuaggaJS / ZXing.js | 浏览器端 JS 库，可以从图片或摄像头实时识别条形码 |
| 后端查询 | 第三方食物数据库 API | 国内：薄荷健康 API、FatSecret API；国外：Open Food Facts、USDA |
| 自建数据库 | 爬取 + 用户贡献 | 初期依赖第三方 API，后期可以自建数据库补充缺失的商品 |

**4. 产品实现方案：**

**方案 A：实时扫描模式**
- 用户点击"扫描条形码"按钮
- 打开摄像头，实时识别条形码（类似支付宝扫一扫）
- 识别成功后立即查询并显示食物信息
- 优点：体验流畅，像扫码支付一样自然
- 缺点：需要摄像头权限，移动端体验更好，Web 端可能有兼容性问题

**方案 B：拍照上传模式**
- 用户拍照上传包装照片
- 后端识别条形码 + 查询数据库
- 返回食物信息
- 优点：实现简单，兼容性好
- 缺点：需要上传图片，速度稍慢

**推荐方案：混合模式**
- 移动端优先使用方案 A（实时扫描）
- Web 端使用方案 B（拍照上传）
- 如果条形码识别失败或数据库查不到，降级到拍照识别（第4点的方案）

**5. 数据库查询失败的处理：**
- 如果第三方 API 查不到这个条形码（小众品牌、地方特产等）
- 提示用户："暂时没有这个商品的数据，要不要拍照识别？"
- 或者引导用户手动输入营养成分（包装上有营养成分表）
- 用户输入后，存入自建数据库，下次其他用户扫到就能直接查到（众包模式）

**6. AI 在条形码扫描中的作用：**
- 条形码识别本身不需要 LLM（用传统 CV 算法即可）
- AI 只在扫描成功后介入：给出个性化专业建议（如"这个零食糖分较高，建议少吃"）
- 条形码模糊/损坏、数据库查不到等场景不走 LLM，走降级方案（拍照识别或手动输入）

### 6. 身体数据追踪的 AI 接入（已确认）

**用户需求：**
- 身体数据追踪部分是否可以接入 AI？怎么接入？
- 用户不清楚怎么测量时，如何辅助测量或告知用户怎么测量？
- 如何少侵入式地和用户交互？
- 核心问题：**如何轻交互，而不是让用户去专门的对话页面？**

**用户决策：**
- 采用 **AI 智能分析报告模块**
- 通过全局分析用户的所有数据（体重、饮食、运动、睡眠等），给用户一个总结报告
- 支持两种形式：报告页面 + 对话模式（通过 Command Bar）

**核心场景：多维度数据的关联分析与洞察**

**触发时机：**
- 用户记录了足够多的多维度数据（体重、饮食、运动、睡眠等）
- 用户主动点击"AI 分析报告"入口
- 或者通过 Command Bar 提问（如"分析一下我最近的健康状况"）

**AI 做什么：**
1. **数据关联分析**：发现数据之间的关联模式
   - 示例："你的数据显示，睡眠不足 6 小时的日子，第二天的饮食热量平均高出 300kcal。建议优先改善睡眠质量。"

2. **趋势解读**：分析趋势并给出洞察
   - 示例："本月体重下降 3kg，腰围减少 4cm，体脂率下降 2%。你的减脂进展很好！但最近一周体重停滞，可能进入平台期。"

3. **目标进展评估**：分析目标完成情况
   - 示例："你本月减重 2kg，按这个速度，3 个月可能只能减 6kg。要调整目标吗？或者我可以帮你优化饮食和运动计划。"

4. **个性化建议**：基于全局数据给出建议
   - 示例："综合你的饮食和运动数据，建议增加蛋白质摄入（目前平均 60g/天，建议 80g/天），并增加力量训练频率。"

**交互方式：**

**方式 1：报告页面**
- 在"数据分析"或"健康报告"页面，有一个"AI 智能分析"入口
- 点击后生成一份结构化的分析报告：
  - 本周/本月数据总结
  - 关键洞察（数据关联、异常发现）
  - 趋势分析（体重、围度、热量摄入等）
  - 目标进展评估
  - 个性化建议
- 报告以卡片形式展示，用户可以展开/收起各个部分

**方式 2：对话模式（Command Bar）**
- 用户通过 Command Bar 提问：
  - "分析一下我最近的健康状况"
  - "我的减脂进展怎么样？"
  - "为什么我的体重停滞了？"
- AI 基于全局数据回答，支持多轮对话追问

**轻交互实现：**
- 不需要专门的对话页面
- AI 分析以**可收起的卡片**形式出现在数据分析页面
- 用户可以通过 **Command Bar** 主动提问
- 关键洞察可以通过**推送通知**或**页面顶部提醒条**主动触达（如"发现你的睡眠质量影响饮食，点击查看详情"）

### 8. Token 消耗管理与付费（已确认）

**用户需求：**
- 如何限制用户使用 AI，防止 token 消耗过多
- 需要和付费能力相结合

**用户决策：**
- 用户不需要知道 token 的概念
- 可以接入广告，通过主动看广告来获取次数
- 需要搭建后台成本控制体系

**成本控制体系设计：**

**1. 后台成本控制层（对用户不可见）**

| 组件 | 功能 | 实现方式 |
|------|------|---------|
| Token 计量系统 | 记录每次 LLM 调用的 token 消耗 | 每次 API 调用后记录：用户 ID、场景类型、输入 tokens、输出 tokens、成本 |
| 成本预算管理 | 设置每个用户的成本上限 | 免费用户：¥2/月，付费用户：¥20/月，超出后降级或限流 |
| 智能路由 | 根据成本优先级选择模型 | 高优先级场景用 qwen-plus，低优先级用 qwen-turbo |
| 缓存系统 | 相同问题复用结果 | 常见问题（如"腰围怎么量"）缓存结果，不重复调用 LLM |
| 异常检测 | 识别异常消耗行为 | 检测短时间内大量调用、恶意刷接口等行为，自动限流 |

**2. 用户端额度体系（用户可见，但不提 token）**

**免费用户：**
- 每天 10 次"AI 解析"额度（包含饮食记录解析、拍照识别等）
- 每天 3 次"AI 对话"额度（Command Bar 复杂对话）
- 额度用完后提示："今日 AI 解析次数已用完，明天 0 点刷新。想继续使用？[看广告获取 +3 次] [升级会员]"

**广告换额度机制：**
- 看一次激励视频广告（15-30 秒）→ 获得 3 次 AI 解析额度
- 每天最多看 5 次广告（最多获得 15 次额度）
- 广告收益可以覆盖部分 token 成本

**4. 后端验证（防作弊）**

**问题：** 前端可以伪造"看完广告"的请求，刷额度怎么办？

**解决方案：** 服务端验证

```
用户点击"看广告"
  ↓
前端请求后端生成广告 token
  ↓
后端生成唯一 token，存入 Redis（5 分钟过期）
  ↓
前端拿到 token，加载广告
  ↓
广告播放完成后，前端带着 token 请求后端
  ↓
后端验证 token 是否存在且未使用
  ↓
验证通过 → 增加额度，标记 token 已使用
验证失败 → 拒绝请求
```

**更严格的方案（推荐）：**
- 使用穿山甲的**服务端回调**功能
- 广告播放完成后，穿山甲服务器直接回调你的后端
- 后端收到回调后才增加额度（前端无法伪造）

**5. 收益计算**

**假设：**
- 激励视频 eCPM = ¥30（千次展示收益）
- 用户看一次广告 = 1 次展示
- 1000 次展示 = ¥30
- 1 次展示 = ¥0.03

**成本对比：**
- 用户看 1 次广告 → 获得 3 次 AI 解析额度
- 3 次 AI 解析的 token 成本 ≈ ¥0.06（假设每次 0.02 元）
- 广告收益 ¥0.03 vs 成本 ¥0.06 → **亏损 50%**

**优化方案：**
- 调整奖励比例：1 次广告 = 2 次额度（而不是 3 次）
- 或者提高广告展示质量，争取更高的 eCPM
- 或者接受亏损，把广告作为"用户留存"的成本

**6. 审核注意事项**

国内广告平台审核很严格，以下情况可能被拒：
- 应用内容涉及医疗健康（需要资质证明）
- 应用未上架应用商店（需要先上架）
- 广告位设置不合理（如强制看广告才能使用核心功能）
- 应用有违规内容（政治、色情、赌博等）

**健康管家 App 的风险：**
- 涉及健康建议，可能需要提供**医疗器械备案**或**互联网医疗资质**
- 建议在应用内加免责声明："本应用仅供参考，不构成医疗建议"

**7. 替代方案（如果广告审核不过）**

- **积分任务**：用户完成任务（如连续打卡 7 天）获得额度
- **邀请奖励**：邀请好友注册获得额度
- **纯付费模式**：不接广告，只做会员付费

**3. 成本优化策略（后台自动执行）**

| 策略 | 说明 | 预期效果 |
|------|------|---------|
| 场景分级 | 核心场景（饮食记录）用好模型，边缘场景（提醒文案）用小模型 | 降低 30% 成本 |

### 2. 健康类 App 特殊资质

健康管家 App 涉及**健康管理、饮食建议、身体数据追踪**，属于健康类应用，需要额外资质：

| 资质 | 是否必须 | 说明 | 办理难度 | 费用 |
|------|---------|------|---------|------|
| **互联网医疗信息服务许可** | 看情况 | 如果提供医疗建议、疾病诊断，必须办理 | 困难 | ¥5000-20000 |
| **医疗器械备案** | 看情况 | 如果 App 用于疾病诊断、治疗，必须办理 | 非常困难 | ¥10000-50000 |
| **食品经营许可证** | 不需要 | 只有卖食品才需要 | - | - |
| **增值电信业务许可（ICP 证）** | 看情况 | 如果有付费会员、广告收入，建议办理 | 中等 | ¥3000-10000 |

**健康管家 App 的判断：**

| 功能 | 是否需要医疗资质 | 原因 |
|------|----------------|------|
| 饮食记录、热量计算 | ❌ 不需要 | 属于工具类功能，不涉及医疗诊断 |
| 体重、围度记录 | ❌ 不需要 | 属于数据记录，不涉及医疗诊断 |
| AI 给出饮食建议（"建议少吃糖"） | ⚠️ 灰色地带 | 如果只是通用建议，不针对疾病，一般不需要；但如果涉及疾病（"糖尿病患者应该..."），需要资质 |
| AI 诊断疾病（"你可能有糖尿病"） | ✅ 必须 | 涉及医疗诊断，必须有互联网医疗许可 |
| 血压、血糖监测 | ✅ 必须 | 涉及医疗数据，可能需要医疗器械备案 |

**结论：**
- **健康管家 App（当前设计）不需要医疗资质**，因为：
  - 只做饮食记录、热量计算、体重追踪
  - AI 建议是通用的健康建议，不针对疾病诊断
  - 不涉及血压、血糖等医疗数据监测

**但需要注意：**
- 在 App 内加**免责声明**："本应用仅供参考，不构成医疗建议，如有健康问题请咨询专业医生"
- AI 建议避免使用"诊断"、"治疗"、"疾病"等医疗术语
- 不要针对具体疾病给建议（如"糖尿病患者应该..."）
